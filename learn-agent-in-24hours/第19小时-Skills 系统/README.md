# 第 19 小时：Skills 系统

## 本章定位

把能力组织成可注册、可组合的技能模块。当 Agent 拥有的工具越来越多时，简单的函数列表已经无法满足管理和选择的需求——你需要一个更高层的抽象来描述"Agent 能做什么"。本章引入 Skill（技能）和 Registry（注册表），将零散的 Tool 函数提升为可发现、可组合、可替换的能力单元。

## 核心概念

### Skill（技能）：比 Tool 更高层的能力抽象

在前面的章节中，我们一直用 Tool（工具）来扩展 Agent 的能力——一个 Tool 就是一个可被调用的函数。这在工具数量较少时运作良好，但随着系统演进，你会遇到几个问题：Tool 没有统一的元信息描述（名称、用途、参数约束），调用方必须知道每个函数的具体签名，工具之间缺乏分类和层级关系。

Skill 是对 Tool 的一层封装和提升。一个 Skill 不仅包含执行逻辑（即底层的 Tool 函数），还携带结构化的元数据：技能名称、功能描述、输入输出规格、适用场景标签等。这使得 Agent 或编排层可以在不了解实现细节的前提下，通过元数据来发现和选择技能。

从设计哲学来看，Tool 面向的是"怎么做"，而 Skill 面向的是"能做什么"。这种抽象层次的提升，使得 Agent 系统具备了自省能力——它可以回答"我现在有哪些能力"这个问题，而不需要硬编码一份能力清单。

### Registry（注册表）模式的设计思想

注册表模式是一种经典的软件设计模式，其核心思想是提供一个集中的注册中心，所有组件在初始化时向注册中心登记自己，需要时再从注册中心按条件查询。在 Skills 系统中，Registry 就是所有技能的统一存储和索引。

采用注册表模式带来几个关键优势。第一，解耦——技能的定义者和使用者不需要直接引用彼此，它们只需要共同认识 Registry。第二，动态性——技能可以在运行时注册和注销，无需重启系统。第三，可发现性——任何模块都可以遍历 Registry 来了解系统当前具备的全部能力。

注册表的实现通常很简单：一个字典（或 Map），以技能名为键、技能对象为值。但关键的设计决策在于注册接口的规范——你是否要求必须提供描述？是否需要声明输入输出 schema？是否允许同名覆盖？这些决定直接影响系统的健壮性和可维护性。

### 技能选择（Selection）策略

当 Agent 面对一个用户请求时，如何从注册表中的众多技能里选出最合适的那个？这就是技能选择的问题。选择策略大致分为两个流派。

基于规则的选择是最直接的方式：通过关键词匹配、正则表达式、或预定义的路由表来映射请求到技能。这种方式确定性高、延迟低、可调试性强，但灵活性有限——如果用户请求的表述方式稍有变化，硬编码的规则就可能失效。

基于 LLM 的选择则将技能列表（包括名称和描述）作为上下文提交给语言模型，由模型判断哪个技能最匹配当前请求。这种方式对自然语言的理解能力强，能处理模糊和多义的请求，但引入了额外的 LLM 调用成本和延迟，且结果具有不确定性。

本章同时实现了两种策略（`pick_skill_with_llm` 和 `pick_skill_mock`），这是一种常见的工程实践：在开发和测试阶段使用确定性的 mock 选择，在生产环境切换为 LLM 选择。这种双轨设计让你可以快速验证流程，而不必每次都消耗 API 调用。

### 模块化（Modularity）的工程价值

模块化是指将系统分解为独立的、可替换的组件，每个组件负责一项明确的职责。在 Skills 系统中，模块化体现在多个层面：每个技能是一个独立模块，可以单独开发、测试和部署；注册表是一个独立模块，不关心技能的具体实现；选择策略也是一个独立模块，可以在不影响技能本身的情况下替换。

模块化的核心价值在于降低变更成本。当你需要新增一个技能时，只需编写技能函数并调用 `register_skill` 注册，其余代码不需要任何修改。当你需要更换选择策略时，只需替换选择模块，技能注册表和技能实现都不受影响。这种"开闭原则"（对扩展开放、对修改关闭）在 Agent 系统中尤为重要，因为能力扩展是最频繁的需求。

### Skill vs Tool 的区别与联系

理解 Skill 和 Tool 的关系是掌握本章的关键。Tool 是原子操作层——"调用某个 API""查询数据库""执行计算"，它关注的是一次具体的函数调用。Skill 是能力语义层——"获取当前时间""统计文章字数""生成话题摘要"，它关注的是一项可识别的业务能力。

一个 Skill 可以包含一个 Tool，也可以编排多个 Tool。例如"撰写调研报告"这个 Skill，可能内部先调用搜索 Tool 收集信息，再调用 LLM Tool 进行总结，最后调用格式化 Tool 生成报告。从外部看，这是一个完整的技能；从内部看，它是多个工具的编排。

### 技能系统的扩展性设计

一个设计良好的技能系统应该能够轻松应对未来的扩展需求。常见的扩展方向包括：技能分组与命名空间（避免不同模块注册同名技能）、技能优先级（当多个技能匹配时的冲突解决）、技能依赖声明（某个技能要求另一个技能必须存在）、以及技能版本管理（同一技能的多个版本共存，按策略路由）。

在实际生产中，技能冲突和选择歧义是最常见的问题。技能冲突发生在两个技能的能力描述高度重叠时，选择层无法可靠区分。解决方案通常是优化技能描述使其更具区分度，或引入优先级机制。选择歧义则发生在用户请求过于笼统时，此时可以引入"澄清"技能——不直接执行，而是反问用户以收窄意图。版本管理方面，常用的做法是在注册表中以"技能名@版本号"作为键，配合路由策略实现灰度发布和回滚。

## 新手最容易卡住的点

- 混淆 Skill 和 Tool 的边界——如果你发现自己把每个函数都叫"技能"，说明你还没有理解抽象层次的差异。回头看看 `register_skill` 注册时附带了哪些元信息，这些元信息就是 Skill 比 Tool 多出来的东西。
- 不理解为什么要有两套选择策略——`pick_skill_mock` 不是偷懒，而是工程实践中的确定性保障。先用 mock 跑通流程，再接入 LLM，是降低调试成本的标准做法。
- 忽略注册表的初始化顺序——如果技能注册发生在技能被调用之后，系统不会报错但会表现异常。检查 `register_skill` 的调用时机是排查问题的第一步。

## 建议动手实验

- 新增一个自定义技能（例如"翻译"或"摘要"），注册到系统中，观察选择策略能否正确路由到它。
- 故意注册两个描述非常相似的技能，测试 LLM 选择策略在歧义场景下的表现。
- 将 `pick_skill_with_llm` 替换为一个基于关键词匹配的纯规则版本，对比两种策略的覆盖率和准确率。
- 尝试在同一个 Skill 内部编排两个步骤（如先获取时间再格式化输出），体会 Skill 与 Tool 的层次差异。
