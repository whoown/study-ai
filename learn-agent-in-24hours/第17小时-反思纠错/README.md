# 第 17 小时：反思纠错

## 本章定位

先生成，再审查，再修订——通过迭代式的反思循环提升 Agent 输出质量。本章引入 Reflection 机制，使 Agent 不再"一次生成定终身"，而是能够对自己的输出进行批判性评估并持续改进，这是从"能用"到"好用"的关键跃迁。

## 核心概念

### Reflection（反思）机制的理论基础

Reflection 的核心思想源自元认知（Metacognition）——对自己的认知过程进行认知。在 Agent 系统中，这意味着 Agent 不仅能完成任务，还能评估自己完成得好不好，并据此改进。

为什么需要反思机制？大语言模型的单次生成往往存在各种问题：信息遗漏、逻辑不连贯、格式不规范、与指令不完全对齐等。这些问题在一次生成中难以完全避免，因为模型的生成过程是自回归的——一旦某个 token 生成方向偏了，后续内容只能在偏离的基础上继续。

反思机制提供了一种"自我纠偏"的能力：生成初稿后，用一个独立的评估过程检视初稿的质量，识别出问题和改进方向，然后基于这些反馈生成改进版本。这个过程可以迭代多次，逐步逼近理想输出。

从系统设计的角度看，反思机制将"生成"和"评估"解耦为两个独立的环节，使得评估标准可以独立于生成过程来定义和调整。这大大增加了系统的可控性。

### Draft → Reflect → Revise 三阶段循环

反思机制的标准工作流包含三个阶段。

**Draft（草稿生成）** 是第一阶段，Agent 根据原始输入生成一个初始回答。这个阶段与普通的 LLM 生成无异，不需要追求完美——生成一个"合理但可能有缺陷"的初稿即可。

**Reflect（反思评估）** 是第二阶段，对初稿进行系统性的审查。反思过程需要回答几个关键问题：初稿是否完整回答了用户的问题？是否存在事实性错误？逻辑是否连贯？格式是否符合要求？有哪些具体的改进建议？反思的输出应该是结构化的评估结果，而非模糊的"好"或"不好"。

**Revise（修订改进）** 是第三阶段，根据反思阶段识别出的问题和改进建议，生成一个新的、更好的版本。修订时不应全部推倒重来，而是针对性地修正反思中指出的问题，保留初稿中好的部分。

这三个阶段构成一个循环——修订后的版本可以再次进入反思阶段，如果仍有问题则继续修订，直到达到终止条件。

### 反思的实现方式：基于规则 vs 基于 LLM

反思的实现方式决定了评估的维度和深度。

**基于规则的反思** 通过预定义的检查规则来评估输出质量。典型的规则包括：长度检查（是否在合理范围内）、格式检查（是否包含必需的结构元素）、关键词检查（是否覆盖了必要的主题）、禁忌词检查（是否包含不应出现的内容）等。规则反思的优势是确定性强、执行快速、结果可解释；局限性在于只能检查形式化的标准，无法评估语义层面的质量。

**基于 LLM 的反思** 用大语言模型作为"评审者"来评估输出。通过精心设计的 prompt，要求模型从多个维度（准确性、完整性、逻辑性、可读性等）对初稿进行评分和点评。LLM 反思的优势是评估维度丰富、能理解语义和上下文；局限性在于评估本身也可能出错（LLM 可能对自己的错误"视而不见"）、且增加了延迟和成本。

最佳实践是两者结合——先用规则快速过滤明显问题（如长度过短、格式缺失），再用 LLM 进行深层语义评估。规则反思作为"硬性门槛"，LLM 反思作为"质量提升"。

### 质量循环（Quality Loop）的终止条件

反思循环不能无限进行，需要合理的终止条件。常见的终止策略包括以下几种。

**固定迭代次数** 是最简单的策略——设定最大循环次数（如 3 次），到达后无论质量如何都停止。优点是可预测、可控；缺点是可能在已经足够好的情况下浪费迭代，或在仍需改进时过早停止。

**质量阈值** 为反思评估设定一个通过阈值——当评估分数超过阈值时停止。这种方式更智能，但前提是评估分数本身是可靠的。

**改进停滞检测** 比较前后两轮迭代的质量差异——如果改进幅度低于某个阈值，说明已经进入了"边际收益递减"阶段，继续迭代意义不大。

**组合策略** 将以上条件通过"与"或"或"逻辑组合，例如"质量达标或迭代 3 次后停止"。这是实践中最常用的方式。

### Self-Refine 论文的核心思想

Self-Refine（Madaan et al., 2023）是反思机制领域的标志性工作。论文的核心发现是：同一个 LLM 可以同时扮演"生成者"和"评审者"两个角色——先生成初稿，然后对自己的初稿提出反馈，再根据反馈改进，整个过程不需要额外的模型或人工标注。

论文在多个任务上验证了这种自我改进的有效性：代码生成、数学推理、文本摘要等。关键发现包括：即使是强大的模型，通过 Self-Refine 也能在初始输出的基础上获得显著提升；改进主要集中在前 2-3 轮迭代，之后收益迅速递减。

Self-Refine 的局限性在于：如果模型在某个维度上存在系统性盲区（比如始终无法识别某类逻辑谬误），那么自我反思也无法发现并修正这类问题。此时引入外部验证器（如代码执行器、事实检查工具）就成为必要的补充。

### 反思机制的常见问题

**无限循环** 是最需要警惕的问题——反思总能找到问题，修订总能做出修改，但修改后又引入新的问题，导致循环永远无法终止。其根源通常是终止条件设计不合理，或反思标准过于严苛。设置最大迭代次数作为兜底是必要的安全措施。

**过度修改** 指在修订阶段对初稿做了不必要的大幅改动，反而破坏了原本正确的部分。修订 prompt 中应明确要求"针对性修改，保留好的部分"，避免每次修订都等于重新生成。

**反思质量退化** 指在多轮迭代后，反思的质量本身开始下降——可能因为上下文变长导致模型注意力分散，或模型开始对自己的输出产生"惯性认同"。控制反思的上下文长度、只保留最新版本而非全部历史，是缓解退化的手段。

### 反思的优化方向

**多维度评估** 将反思拆分为多个独立的评估维度（准确性、完整性、可读性、格式等），每个维度独立打分和评论。这比笼统的"总体评价"更精确，也让修订阶段能有针对性地改进特定维度。

**外部验证器** 引入非 LLM 的验证手段来增强反思的可靠性。例如：用代码执行器验证生成代码的正确性、用搜索引擎验证事实性声明、用格式化工具验证输出格式。外部验证器的结果是确定性的，能弥补 LLM 自我评估的不确定性。

**分角色反思** 让不同的"角色"从不同视角评估同一份输出——一个角色关注事实准确性，另一个关注用户体验，第三个关注安全合规。这种方式能覆盖更广的评估面，减少单一视角的盲区。

## 新手最容易卡住的点

- 把 `rule_reflect` 和 `llm_reflect` 的输出直接当作"正确答案"——反思本身也可能出错，需要设计对反思结果的可信度评估。
- 在 `run_reflection_loop` 中不设上限，导致循环次数失控——始终确保有最大迭代次数作为兜底。
- 修订时不传入反思意见，而是让模型"凭空改进"——修订的质量高度依赖于反思阶段提供的具体、可操作的改进建议。
- 期望每一轮反思都能带来显著提升——实际上改进主要集中在前 1-2 轮，后续轮次通常只是微调。

## 建议动手实验

- 对比 `rule_reflect` 和 `llm_reflect` 对同一份初稿的评估结果，观察规则检查和语义评估的差异。
- 修改 `run_reflection_loop` 的最大迭代次数，观察不同轮次下最终输出质量的变化曲线。
- 故意让 `draft_answer` 生成一个质量较差的初稿（如截断输出），观察反思修订循环能否将其修复到可接受的水平。
- 在 `llm_reflect` 的 prompt 中增加新的评估维度，观察修订方向的变化。
