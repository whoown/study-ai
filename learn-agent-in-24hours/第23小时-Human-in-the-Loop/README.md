# 第 23 小时：Human-in-the-Loop

## 本章定位

把人工审批纳入自动流程。自动化并不意味着完全无人干预——在高风险决策、敏感操作或质量要求极高的场景中，人类的判断仍然不可替代。本章通过构建一个包含提案生成、人工审批和条件执行的完整流程，展示如何在 Agent 自动化管线中优雅地嵌入人类决策节点。

## 核心概念

### Human-in-the-Loop（HITL）的必要性与适用场景

Human-in-the-Loop（人在回路中）是指在自动化流程的关键决策点引入人类参与者，由人类做出最终判断后流程才继续推进。这不是对自动化的否定，而是对自动化边界的清醒认知。

HITL 的必要性来自两个方面。第一是能力边界——当前的 LLM 在事实准确性、数值计算、法律合规判断等方面仍有局限，某些决策确实需要人类的专业知识。第二是责任边界——即使 AI 能做出正确判断，某些操作（如资金转账、内容发布、系统变更）在组织治理层面要求有人类签字确认。

HITL 的典型适用场景包括：涉及金钱或合同的操作、面向公众的内容发布、不可逆的系统变更（如数据删除、配置推送）、以及需要领域专家判断的分析结论。反过来，低风险、高频次、可回滚的操作（如日志归档、缓存清理）通常不需要 HITL。

### 审批（Approval）流程的设计模式

审批流程的核心是将一个连续的自动化链路"切断"为两段：审批前的准备阶段和审批后的执行阶段。准备阶段负责收集信息、生成提案；审批节点暂停流程等待人类决策；执行阶段根据审批结果决定是继续还是终止。

审批流程的设计模式大致有三种。第一种是"同步阻塞"——流程在审批节点暂停，直到人类响应后才继续。这是最简单的实现，本章采用的就是这种方式（`prompt_human_approval` 函数）。它的局限是流程会一直占用资源等待。

第二种是"异步回调"——流程在审批节点将状态持久化后挂起，释放计算资源。当人类审批完成后，系统通过回调机制恢复流程执行。这种方式资源效率更高，适合审批可能需要数小时甚至数天的场景。

第三种是"审批队列"——所有待审批的请求进入一个集中的队列，审批者可以批量查看和处理。这种方式对审批者更友好，特别是当系统每天产生大量审批请求时。

### 事件（Event）驱动的 HITL 架构

本章使用事件驱动的方式记录 HITL 流程中的每一个关键动作（`emit_hitl_event`）。事件驱动架构在这里的价值在于：它将"发生了什么"（事实记录）与"接下来做什么"（业务逻辑）解耦。

每个事件包含时间戳、事件类型、相关数据和触发者信息。典型的事件序列是：`proposal_created`（提案生成）→ `approval_requested`（请求审批）→ `approval_granted` 或 `approval_denied`（审批结果）→ `action_executed` 或 `action_cancelled`（最终执行）。

事件流的价值远超过日志。它可以用于审计追踪（谁在什么时候批准了什么）、流程分析（审批平均等待多长时间？拒绝率是多少？）、以及系统恢复（如果系统崩溃，可以从最后一个事件重放恢复到正确状态）。在合规性要求高的行业（金融、医疗），审计追踪不是可选的而是必须的。

### 安全性（Safety）考量

HITL 的核心目的之一就是安全。但"安全"是一个多层次的概念，不同层次需要不同的保障机制。

操作安全是最基本的层次：确保 Agent 不会执行超出授权范围的操作。实现方式是为每类操作标记风险等级，高风险操作必须经过人工审批，低风险操作可以自动执行。

数据安全关注的是敏感信息的保护。审批提案中可能包含用户数据、商业机密或个人隐私。提案展示时需要进行脱敏处理，审批日志也需要控制访问权限。

决策安全是最高层次的考量：即使审批者同意了，这个决策本身是否合理？好的 HITL 系统会在提案中提供充分的上下文信息和风险提示，帮助审批者做出知情决策，而不是简单地抛出一个"同意/拒绝"的选择。

### HITL 的粒度选择

HITL 的粒度——即在多细的层面要求人工介入——是一个重要的设计决策。

全量审批意味着每一步操作都需要人类确认。这最安全，但严重拖慢了流程，且审批者会因为"审批疲劳"而降低审核质量——当你每天要审批 200 个请求时，到后面几乎就是无脑点"同意"。

风险审批是更实用的策略：只在风险超过阈值时要求审批。风险评估可以基于规则（涉及金额超过 X 元）、基于模型（AI 评估操作风险分数）、或基于历史（该类型操作历史出错率高于 Y%）。这种方式在安全性和效率之间取得了平衡。

还有一种"抽检审批"模式：大部分请求自动通过，但随机抽取一定比例进行人工审核。这不是为了阻止具体的高风险操作，而是为了持续评估自动化系统的整体质量，发现系统性的偏差。

### 异步审批与超时处理

在真实场景中，审批者不会时刻守在电脑前等待审批请求。从请求发出到审批完成，可能经过数分钟到数天不等。这就需要异步审批机制。

异步审批的实现需要解决几个问题：状态持久化（流程状态必须保存到持久存储，而不是保留在内存中）、通知机制（通过邮件、即时通讯、短信等渠道通知审批者）、以及恢复机制（审批完成后如何唤醒挂起的流程）。

超时处理是异步审批必须考虑的问题。如果审批者迟迟不响应怎么办？常见的策略包括：设置超时时间后自动拒绝（安全优先）、设置超时后自动通过（效率优先，仅限低风险操作）、超时后升级到上级审批者、以及超时后发送催办通知并延长等待。选择哪种策略取决于业务场景的风险偏好。

### HITL 的用户体验设计

HITL 不仅是技术问题，也是用户体验问题。审批者的体验直接影响审批质量和响应速度。

好的审批界面应该做到：信息充分但不冗余（审批者能快速理解需要审批什么、为什么需要审批、风险是什么）、操作简单（一键同意/拒绝，支持批量操作）、上下文可追溯（可以查看完整的操作历史和相关数据）。

本章中 `present_for_approval` 函数的设计体现了"信息充分"这一原则——它不是简单地问"同意吗？"，而是先展示完整的提案内容，让审批者在充分了解情况后做出判断。

### 常见问题：审批瓶颈与流程中断恢复

审批瓶颈是 HITL 系统最普遍的问题。当自动化部分处理速度远快于人类审批速度时，待审批队列会快速堆积，导致整个系统的吞吐量被审批环节限制。缓解策略包括：并行审批（多个审批者处理不同请求）、审批委托（审批者可以将权限临时委托给他人）、以及智能路由（将审批请求分配给当前负载最低的审批者）。

流程中断恢复是另一个关键问题。如果系统在审批等待过程中崩溃，如何从中断点恢复？答案是事件溯源——通过重放事件序列，系统可以恢复到崩溃前的状态，然后继续等待审批或处理已到达的审批结果。这也是本章强调事件记录（`emit_hitl_event`）的原因之一。

## 新手最容易卡住的点

- 将 HITL 简单理解为"加一个 input() 暂停"——真正的 HITL 涉及提案生成、信息展示、状态持久化和事件记录，`prompt_human_approval` 只是冰山一角。
- 忽略审批拒绝后的处理逻辑——审批被拒绝不等于流程结束，可能需要回退到提案阶段重新修改。检查 `finalize_release` 在审批通过和拒绝时的不同行为。
- 没有区分"需要审批的操作"和"不需要审批的操作"——如果所有操作都要审批，系统会变得不可用。思考哪些操作适合自动执行，哪些必须人工确认。

## 建议动手实验

- 修改 `draft_proposal` 的提案内容，加入风险等级字段，然后在审批逻辑中根据风险等级决定是否需要人工审批。
- 实现一个简单的超时机制：如果审批者在 30 秒内没有响应，自动拒绝并记录超时事件。
- 在审批拒绝后，实现"带修改意见回退"逻辑——让系统根据审批者的反馈重新生成提案。
- 收集所有 `emit_hitl_event` 产生的事件，汇总输出一份流程审计报告，包含每个步骤的执行时间和审批结果。
