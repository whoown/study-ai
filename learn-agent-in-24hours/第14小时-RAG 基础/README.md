# 第 14 小时：RAG 基础

## 本章定位

把检索结果接回生成流程，构建完整的 RAG 管线。在上一章掌握了 Embedding 和向量检索之后，本章将这些能力串联起来：从原始文本的分块与索引构建，到检索、上下文组装、最终生成，走通"检索增强生成"的完整链路。

## 核心概念

### RAG 的核心思想与为什么需要它

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思想是：在大语言模型生成回答之前，先从外部知识库中检索相关信息，将其作为上下文注入 prompt，让模型基于这些"证据"来生成回答。

为什么需要 RAG？大语言模型存在三个根本性限制。第一，知识截止——模型的训练数据有时间边界，无法回答训练数据之后的问题。第二，知识容量——即使是最大的模型也无法记住所有领域知识，对长尾知识的覆盖不足。第三，幻觉——模型在不确定时倾向于"编造"看起来合理的答案。RAG 通过引入外部检索来缓解这三个问题：知识库可以实时更新、容量不受模型参数限制、生成时有明确的"证据"支撑。

与微调（Fine-tuning）相比，RAG 的优势在于不需要重新训练模型，知识更新只需更新知识库即可。但 RAG 也引入了新的复杂度：检索质量直接决定生成质量，整个系统的链路更长、出错点更多。

### 文本分块（Chunking）策略

在构建向量索引之前，需要将长文档切分为较小的文本块（chunk）。分块的质量直接影响后续检索的精度。

**固定长度分块** 是最简单的策略——按字符数或 token 数将文本切为等长片段。优点是实现简单、可预测；缺点是可能在语义不完整的位置截断，导致单个 chunk 缺少必要的上下文。

**语义分块** 尝试在自然的语义边界处切分，比如段落边界、章节标题、主题切换点等。这种方式保留了更完整的语义单元，但实现复杂度更高，通常需要额外的模型来判断语义边界。

**递归分块** 是 LangChain 等框架采用的策略——先按大的分隔符（如双换行）切分，如果某段仍然过长，再用更小的分隔符（如单换行、句号）继续切分，直到满足长度要求。这种方式在保持语义完整性和控制长度之间取得了较好的平衡。

分块时还需考虑 **chunk overlap（重叠）**：相邻块之间共享一部分文本，避免关键信息恰好落在两块的边界上而被"切断"。典型的重叠率为 chunk 长度的 10%-20%。

### 检索（Retrieve）阶段：稀疏检索 vs 密集检索 vs 混合检索

检索的目标是从大量文本块中快速找出与用户查询最相关的 top-K 个结果。

**稀疏检索** 基于词频统计，代表方法是 BM25。它通过关键词匹配来衡量相关性，优势是速度快、可解释性强、对精确关键词匹配表现好；劣势是无法处理同义词和语义改写。

**密集检索** 基于 Embedding 向量，将查询和文档都编码为向量后通过相似度计算来排序。优势是能捕捉语义相似性，即使措辞完全不同也能找到相关内容；劣势是对精确匹配（如专有名词、编号）的处理不如稀疏检索。

**混合检索** 同时使用稀疏和密集两种方式，将两者的分数融合后排序。这是目前工业界的最佳实践——既能享受语义理解的优势，又不丢失关键词精确匹配的能力。融合方式通常采用倒数排名融合（RRF）或加权线性组合。

### 增强（Augment）：如何将检索结果组织进 Prompt

检索到相关文本块后，需要将它们有效地组织进 prompt，让大语言模型能够基于这些上下文来生成回答。这个环节看似简单，实则有不少设计考量。

首先是 **上下文窗口管理**。模型的上下文窗口有限，如果检索到的内容过多，需要取舍。常见策略是按相关性排序后截断，或者对检索结果做摘要压缩。

其次是 **prompt 模板设计**。典型的 RAG prompt 包含三部分：系统指令（告诉模型基于上下文回答、不要编造）、检索到的上下文、用户问题。上下文的呈现方式也很重要——是否标注来源、是否编号、是否按相关性排序，都会影响模型的回答质量。

最后是 **指令引导**。需要在 prompt 中明确要求模型"只根据提供的上下文回答"，并在信息不足时坦诚告知。这有助于减少幻觉，但不能完全消除。

### RAG 的完整 Pipeline

一个典型的 RAG 系统包含两个阶段：离线索引阶段和在线查询阶段。

**离线索引阶段**：原始文档 → 文本提取 → 分块 → Embedding 编码 → 存入向量数据库。这个过程通常只需执行一次（或在文档更新时增量执行）。

**在线查询阶段**：用户问题 → Embedding 编码 → 向量检索 → （可选）重排序 → Prompt 组装 → LLM 生成 → 返回回答。

其中 **重排序（Re-ranking）** 是一个常见的优化步骤。初始检索通常基于向量相似度快速召回较多候选（如 top-20），然后用一个更精细的交叉编码模型（Cross-Encoder）对候选重新打分排序，取最终的 top-K（如 top-5）。这种"粗筛 + 精排"的两阶段策略能显著提升检索精度。

### RAG 的常见问题

**检索不准** 是最核心的问题——如果检索到的内容与问题不相关，模型要么答非所问，要么回退到自身的参数知识（可能产生幻觉）。原因可能是分块粒度不合理、Embedding 模型对目标领域适配不足、或查询本身过于模糊。

**上下文过长** 会导致模型"迷失在中间"（Lost in the Middle）——研究表明模型更关注上下文的开头和结尾，对中间部分的利用率下降。控制上下文长度和合理排列检索结果的顺序是缓解手段。

**幻觉仍然存在**。即使提供了正确的上下文，模型仍可能生成与上下文不一致的内容。这需要在 prompt 设计和后处理中加入额外的验证机制。

### RAG 的优化方向

**Query Rewriting（查询改写）** 在检索前对用户原始问题进行改写或扩展，使其更适合检索。例如将口语化的问题改写为关键词丰富的检索查询，或拆分复杂问题为多个子查询分别检索。

**Re-ranking（重排序）** 通过引入交叉编码模型对初始检索结果进行精排，显著提升最终送入 prompt 的文本块的质量。

**Chunk Overlap（分块重叠）** 在相邻 chunk 之间保留重叠部分，确保跨边界的信息不会丢失。配合合理的 chunk 大小设计，可以有效提升检索的召回率。

其他优化还包括：多向量检索（为同一文本生成多个不同角度的向量）、自适应 chunk 大小、检索结果的去重与合并等。

## 新手最容易卡住的点

- 分块粒度把握不好——chunk 太大导致检索结果混杂无关信息，太小导致上下文不完整。建议从 500-1000 字符起步，根据实际效果调整。
- 忽略 `build_rag_prompt` 的设计重要性——prompt 模板对最终回答质量的影响不亚于检索本身。
- 将"检索到了结果"等同于"检索到了正确结果"——需要关注检索结果的相关性评估，而非仅仅确认返回了内容。
- 没有区分离线索引和在线查询两个阶段，导致对性能瓶颈的判断出错。

## 建议动手实验

- 修改 `chunk_text` 中的分块大小参数，观察不同粒度下检索结果的变化。
- 对比 `memory_retrieve` 和 `chroma_retrieve` 在相同查询上的检索结果差异，理解不同检索后端的行为。
- 在 `build_rag_prompt` 中调整上下文的组织方式（如改变顺序、增减检索结果数量），观察 `generate_answer` 输出的变化。
- 构造一个知识库中没有答案的问题，观察系统是否会产生幻觉回答。
