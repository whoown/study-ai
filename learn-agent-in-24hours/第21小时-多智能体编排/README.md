# 第 21 小时：多智能体编排

## 本章定位

让 Supervisor 调度多个角色节点。上一章的流水线模式只能线性推进，无法实现"发现问题后回退重做"或"根据中间结果动态调整执行路径"。本章引入 Supervisor（主管）模式和图编排，让一个中心调度节点根据当前状态动态决定下一步该让谁工作，从而支持循环、分支和条件终止。

## 核心概念

### Supervisor（主管）模式的核心思想

Supervisor 模式是多智能体编排中最常见的中心化架构。其核心思想是：设立一个专门的"主管"Agent，它不直接执行具体任务，而是负责观察当前状态、决定下一步由哪个角色执行、并判断何时终止整个流程。

与流水线模式的本质差异在于"决策权"的归属。流水线中，执行顺序在代码中硬编码，运行时不可改变；Supervisor 模式中，执行顺序由 Supervisor 在运行时根据状态动态决定。这意味着系统获得了"适应性"——如果 Analyst 的分析结果不够深入，Supervisor 可以把任务再次路由回 Analyst，而不是盲目推进到 Writer。

Supervisor 本身的能力要求也与工作节点不同。工作节点需要"深度"——在自己的专业领域做到最好；Supervisor 需要"广度"——理解所有角色的能力边界，并做出全局最优的调度决策。因此在实践中，Supervisor 通常使用能力较强的模型，而工作节点可以使用更经济的模型。

### 路由（Routing）决策

路由是 Supervisor 的核心能力——决定下一步交给谁。路由策略可以分为静态路由和动态路由两种。

静态路由是基于预定义规则的路由：如果当前步骤是分析完成，则下一步是写作；如果写作完成，则下一步是评审。静态路由的执行路径在编码时就已确定，虽然不灵活但确定性高，适合流程固定的场景。

动态路由则让 Supervisor 根据运行时状态自主决策。通常的实现方式是将当前状态（已完成的步骤、中间产出、迭代次数等）提交给 LLM，由 LLM 输出下一个应该执行的角色名称。本章的 `_route_supervisor` 就是路由决策的实现。动态路由的优势在于灵活——它可以处理规划阶段无法预见的情况，但也带来了不确定性风险。

实际项目中常见的做法是"受约束的动态路由"：Supervisor 可以自由选择下一个角色，但必须遵守一些硬约束（如最大迭代次数、必须经过的环节、禁止的路由路径等）。这种方式在灵活性和可控性之间取得了平衡。

### 共享状态（Shared State）的管理

在 Supervisor 模式中，所有角色节点需要在一个共享状态上读写。共享状态通常是一个字典或类对象，包含任务描述、各角色的中间产出、当前步骤标记、迭代计数等信息。

共享状态的设计有几个关键考量。首先是状态的结构化程度：松散的字典结构灵活但容易出错（如键名拼写错误），强类型的数据类安全但修改起来较麻烦。其次是状态的更新策略：每个节点是直接修改全局状态（in-place mutation），还是返回状态增量（delta）由编排层统一合并？增量合并的方式更安全，可以支持回滚和审计，但实现复杂度更高。

状态膨胀是生产中的常见问题。随着迭代轮次增加，中间产出不断累积，状态体积会快速增长。这不仅增加内存占用，还会导致传递给 LLM 的上下文过长。常见的应对策略包括：只在状态中保留最近 N 轮的产出、对历史产出做摘要压缩、以及将大块数据存储在外部（如数据库）并只在状态中保留引用。

### 图编排（Graph）在多智能体中的应用

图是表达复杂编排逻辑的自然数据结构。在图编排中，每个角色节点是图中的一个节点，节点之间的有向边表示可能的流转关系，路由函数决定在运行时沿哪条边前进。

本章的 `build_graph` 函数定义了节点集合、边的连接关系以及路由规则。这种声明式的图定义有几个优势：第一，流程结构一目了然，可以直接从图定义中读出所有可能的执行路径；第二，图结构可以被序列化和可视化，方便调试和文档化；第三，图是 LangGraph 等编排框架的基础抽象，掌握图编排的思维方式可以无缝迁移到生产级框架。

图编排相对于纯代码编排（if-else 或函数调用链）的核心优势在于"可分析性"——你可以对图做静态分析：是否存在不可达节点？是否存在没有终止条件的循环？入度为零的节点是否都是合法的起始点？这些分析在复杂系统中非常有价值。

### Supervisor vs 去中心化编排的对比

除了 Supervisor 这种中心化模式，还有去中心化的编排方式——每个 Agent 自己决定下一步交给谁。例如 Analyst 完成分析后，自己决定是交给 Writer 还是交给另一个 Analyst 补充。

去中心化模式的优势是没有单点瓶颈，每个 Agent 可以并行决策。但它的问题也很明显：缺乏全局视野（每个 Agent 只看到自己的局部状态），容易出现活锁（两个 Agent 互相推诿），且难以施加全局约束（如最大迭代次数）。

实践中的建议是：当角色数量较少（3-5 个）且流程清晰时，优先使用 Supervisor 模式；当系统规模很大（数十个 Agent）且各 Agent 高度独立时，可以考虑去中心化或混合模式。本章的规模适合使用 Supervisor 模式。

### 编排循环的终止条件设计

循环是 Supervisor 模式最强大也最危险的特性。Supervisor 可以将任务反复路由给同一个角色以迭代优化，但如果终止条件设计不当，就会陷入无限循环。

常见的终止条件包括：达到最大迭代次数（硬性上限，防止失控）、Supervisor 显式输出"完成"信号（基于对产出质量的判断）、所有必要步骤均已完成（基于检查清单）、以及总 token 消耗达到预算上限（成本控制）。

稳健的设计通常采用多重终止条件的组合：首先检查是否达到硬性上限（迭代次数、token 预算），如果没有则让 Supervisor 判断是否满意当前产出，如果 Supervisor 判断继续则检查是否有实质性进展（防止无效循环）。任何一个条件触发都会终止循环。

### 常见问题：调度死循环与上下文丢失

调度死循环是最常被报告的问题。典型场景是：Supervisor 认为 Analyst 的产出不够好，路由回 Analyst；Analyst 产出相同的结果（因为它没有额外信息可用）；Supervisor 再次判断不够好，再次路由回 Analyst——如此往复。解决方案是在状态中记录迭代历史，并在 Supervisor 的提示词中明确告知"如果同一角色已经执行了 N 次且没有改善，应终止或换一种策略"。

角色间上下文丢失是另一个常见问题。由于共享状态通常只传递最新的产出，早期角色产生的关键信息可能在多轮迭代后被覆盖。解决方案是在状态中维护一个 append-only 的历史记录，或者为关键信息设置独立的状态字段（不会被后续步骤覆盖）。

## 新手最容易卡住的点

- 不理解 Supervisor 与工作节点的职责区别——Supervisor 不是"更强的 Agent"，而是"不做事只调度的管理者"。如果你的 Supervisor 在执行具体任务，说明角色划分有问题。
- 路由逻辑中的状态依赖——`_route_supervisor` 根据状态中的哪些字段做决策？修改这些字段的值，观察路由行为的变化，是理解动态路由的最佳方式。
- 忽略终止条件——如果运行时程序"卡住"不结束，首先检查终止条件是否可达，其次检查 Supervisor 的路由决策是否陷入了循环。

## 建议动手实验

- 在 `_route_supervisor` 中加入打印语句，输出每次路由决策的依据和结果，观察 Supervisor 的决策过程。
- 故意去掉最大迭代次数限制，观察系统是否会陷入死循环，以及在什么条件下会陷入。
- 新增一个 Fact-Checker 角色节点，让 Supervisor 可以在 Writer 产出后路由到它进行事实核查，体会图编排的扩展灵活性。
- 对比将 Supervisor 的路由逻辑从动态（LLM 决策）改为静态（固定顺序）后，系统行为和输出质量的差异。
