# 第 1 小时：理解最小 Agent

## 本章定位

用纯 Python 规则搭出 Agent 最小闭环——不依赖任何大模型，仅靠硬编码的 `fake_brain` 模拟决策，配合 `tool_multiply`/`tool_say` 两个工具和 `run_episode` 驱动循环，让你在最简单的环境中看清 Agent 的骨架结构。理解这个骨架之后，后续替换真实 LLM、接入更复杂的工具链才不会迷失方向。

## 核心概念

### Agent 的定义与本质

Agent（智能体）并不是一个模糊的营销概念，而是一种具有明确工程结构的程序范式。其本质可以用三个词概括：**感知—决策—行动**。Agent 持续感知外部环境的状态，基于某种策略做出决策，然后执行相应的行动去改变环境，再感知新的状态——如此往复形成闭环。

这个定义和传统的"输入—处理—输出"程序有本质区别。传统程序执行一次就结束了，而 Agent 是一个持续运转的循环体。它不断与环境交互，每次行动的结果会影响下一次决策。这意味着 Agent 天然具备处理多步骤任务的能力——它不需要在一次调用中完成所有事情，而是可以逐步推进、观察反馈、调整策略。

在本章中，`fake_brain` 就是"决策"的载体。虽然它只是硬编码的 if-else 规则，但它扮演的角色和后续的大语言模型完全一致：接收当前状态信息，返回下一步应该做什么。这种设计让你在不依赖外部服务的情况下理解 Agent 的完整运作机制。

从工程角度看，Agent 区别于普通脚本的关键在于**自主性**：它能根据运行时的实际情况动态调整行为路径，而不是沿着预设的线性流程机械执行。即使是最简单的规则 Agent，也已经具备了这种特征。

### Tool（工具）的抽象

工具是 Agent 与外部世界交互的手段。Agent 本身只负责"思考"和"决策"，真正的"做事"——无论是计算、查询数据库、调用 API 还是发送消息——都通过工具完成。这种分离是 Agent 架构的核心设计原则之一。

为什么需要工具抽象？因为 Agent 的决策层（无论是规则还是 LLM）天然不擅长精确计算、实时数据获取等操作。让 LLM 去做两个大数的乘法，结果大概率是错的；但让它决定"现在需要做乘法"，然后把计算交给专用函数，准确率就是 100%。工具抽象让 Agent 的每个部分都做自己最擅长的事。

一个好的工具设计有几个特征：**明确的输入输出契约**、**无副作用或副作用可预期**、**统一的调用接口**。本章中 `run_tool` 就是统一调用入口的体现——所有工具通过同一个分发函数调用，便于做参数校验、异常处理和日志记录。这种模式在后续引入更多工具时会体现出巨大的工程价值。

在实际开发中，工具的粒度设计是一个需要权衡的问题。工具太粗（一个工具做太多事），Agent 的灵活性会降低；工具太细（每个小操作都是一个工具），Agent 的决策负担会增大。通常的经验是：**一个工具对应一个原子操作**，即一个有明确业务含义的最小动作。

### Agent Loop（循环）的工作原理

Agent Loop 是 Agent 的心跳。它是一个 while 循环，每一轮执行固定的步骤序列：获取当前状态 → 调用决策层 → 解析决策结果 → 执行工具 → 收集观察结果 → 更新状态 → 判断是否结束。

这个循环结构之所以重要，是因为它将 Agent 的复杂行为分解为可预测、可调试的离散步骤。每一轮循环都是独立的、可观测的。你可以在任意一轮打断它、检查状态、甚至修改状态后继续执行。这对于调试和理解 Agent 行为至关重要。

循环的终止条件是设计中容易被忽视的关键点。常见的终止方式包括：决策层返回"完成"信号、达到最大轮次上限、出现不可恢复的错误。本章的 `run_episode` 通过检查 `fake_brain` 的返回来判断是否结束。在后续章节中，终止条件会变得更加多样和复杂，但基本模式始终一致。

循环的最大轮次限制（max steps）是一个重要的安全机制。如果 Agent 的决策层出了问题（比如 LLM 陷入了重复输出），没有轮次限制的循环会无限运行下去，消耗大量资源甚至造成事故。这是 Agent 工程中最基本的防御措施之一。

### Observation（观察）在闭环中的作用

Observation 是 Agent 闭环中最容易被忽视的环节，但它恰恰是让 Agent 具备"智能"的关键。Observation 是工具执行后返回给决策层的反馈信息，它告诉 Agent "你刚才那个动作产生了什么效果"。

没有 Observation，Agent 就是一个开环系统——它只管发出指令，却不知道指令的执行结果。这就像闭着眼睛走路：你可以迈步，但无法根据地形调整方向。Observation 让 Agent 从开环变成闭环，从"盲目执行"变成"感知—反应"。

在本章的实现中，Observation 的流转路径很清晰：工具函数执行后返回结果 → 结果被格式化为 Observation → Observation 被传回给决策层（`fake_brain`）→ 决策层根据 Observation 决定下一步。这个链路在后续所有章节中都是一致的，只是决策层从规则变成了 LLM，Observation 的格式从简单字符串变成了结构化数据。

Observation 的质量直接影响 Agent 的表现。如果工具只返回"成功"而不返回具体结果，Agent 就无法基于结果做后续推理。设计 Observation 时应该包含足够的信息让决策层做出合理的下一步判断，但也不应该包含过多无关信息造成干扰。

## 新手最容易卡住的点

- **混淆 Agent 和普通脚本**：Agent 的核心是循环和动态决策，不是从头到尾执行一遍。如果你发现自己在想"这不就是个函数调用吗"，说明你还没有注意到循环结构带来的本质区别。
- **不理解 fake_brain 的意义**：它不是"简陋的替代品"，而是一种刻意的教学设计。规则引擎和 LLM 在 Agent 架构中占据的是同一个位置，只是决策能力不同。
- **忽略 Observation 的流转**：只关注工具被调用了，不关注工具的返回值是如何被传回决策层的。Observation 的流转路径是理解闭环的关键。
- **认为 run_tool 是多余的封装**：统一工具入口在工具少的时候看不出价值，但它是后续扩展（错误处理、日志、权限控制）的基础。

## 建议动手实验

- 在 `fake_brain` 中添加一条新规则，让 Agent 面对特定输入时执行不同的工具序列，观察循环如何响应。
- 新增一个工具函数（比如 `tool_add`），注册到 `run_tool` 中，然后修改 `fake_brain` 让它在某些情况下调用这个新工具。
- 在 `run_episode` 的循环中加入打印语句，输出每一轮的决策和 Observation，追踪完整的闭环流转。
- 把 `run_episode` 的最大轮次设为 1，观察 Agent 行为的变化，理解循环长度对 Agent 能力的影响。
