# 第 5 小时：多工具决策

## 本章定位

让模型在多个工具之间做出选择，并能将多个工具调用串联起来完成复杂任务。前几章的 Agent 只面对 1-2 个工具，选择空间很小；本章引入更多工具（如 `tool_get_unit_price`、`tool_multiply`、`tool_say`、`tool_finish`），迫使模型真正进行"决策"——在不同工具中挑选最适合当前步骤的那一个，并规划调用顺序。

## 核心概念

### Tool Selection：模型如何在多工具中选择

当 Agent 可用的工具数量从 2 个增长到 5 个、10 个甚至更多时，"选择正确的工具"本身就成为了一个需要推理的问题。这就是 Tool Selection——模型根据当前上下文和任务目标，从可用工具集中挑选最合适的工具来执行。

模型进行 Tool Selection 的依据主要有三个：**工具的 description**（工具能做什么）、**当前对话上下文**（用户问了什么、之前做了什么）、**任务目标**（最终要达成什么结果）。模型在内部进行语义匹配——将当前需要完成的操作与各工具的描述进行对比，选择匹配度最高的工具。

Tool Selection 的质量直接受到工具集设计的影响。如果两个工具的功能描述过于相似，模型就难以区分它们的适用场景。例如，如果同时存在 `search_web` 和 `search_database` 两个工具，但 description 都只写"搜索信息"，模型就会频繁选错。好的工具集设计应当保证每个工具有明确的、可区分的职责边界。

工具数量也影响 Selection 质量。研究表明，当工具数量超过 15-20 个时，大多数模型的选择准确率会显著下降。应对这个问题的常见策略包括：**工具分组**（根据任务类型动态加载相关工具子集）、**工具推荐**（先用一个简单模型筛选候选工具，再让主模型从少量候选中选择）、**层级工具**（用"元工具"代表一组功能，模型先选择类别再选择具体工具）。

### Tool Chaining：工具调用链的设计

现实中的任务很少能通过单次工具调用完成。大多数有意义的任务需要多个工具按特定顺序协作——这就是 Tool Chaining。例如，"计算 5 箱苹果的总价"需要先调用 `get_unit_price` 获取单价，再调用 `multiply` 计算总价，最后调用 `say` 输出结果。

Tool Chaining 的核心挑战是**数据流传递**：前一个工具的输出往往是后一个工具的输入。在 Agent 架构中，这种传递通过 Observation 机制实现——工具 A 的结果作为 Observation 进入上下文，模型在下一轮推理时读取这个 Observation，从中提取所需数据传给工具 B。

这种基于上下文的隐式数据传递既是优势也是风险。优势在于灵活性：模型可以自由决定如何组合和传递数据，不需要预先定义严格的工具间接口。风险在于不可靠性：模型可能在长链中"忘记"早期的 Observation，或者错误地理解数据格式，导致传递的参数有误。

设计 Tool Chaining 时的一个重要原则是**保持工具的原子性**。每个工具应该完成一个独立的、明确的操作，而不是试图在一个工具中包含多个步骤。原子性的工具更容易被模型正确理解和组合，也更容易在不同任务中复用。如果你发现一个工具需要内部做多步操作，考虑将它拆分为多个更小的工具。

链式调用的可靠性还与链的长度正相关：链越长，中间出错的概率越高。在实践中，超过 5-6 步的工具链往往需要额外的错误恢复机制或中间检查点。

### 上下文对工具选择的影响

上下文（Context）在多工具决策中扮演着核心角色。同一个用户请求，在不同的上下文下可能需要调用完全不同的工具。例如，"查一下价格"——如果前文在讨论苹果，应该调用商品查价工具；如果前文在讨论股票，应该调用行情查询工具。

上下文对工具选择的影响主要通过消息历史（messages）传递。Agent 循环中累积的每一条消息——用户输入、模型推理、工具返回——都构成了当前的上下文。模型在做 Tool Selection 时，不仅看当前这一条消息，而是综合所有历史消息来做判断。

这带来了一个重要的工程问题：**上下文污染**。当对话历史过长或包含无关信息时，模型可能被噪声干扰而做出错误选择。常见的应对策略包括：及时清理已完成任务的上下文、使用摘要替代详细历史、在关键决策点重申任务目标等。

上下文窗口的有限性也要求开发者做出权衡。保留更多的历史可以让模型做出更"知情"的决策，但也增加了 token 消耗和推理延迟。在多工具场景中，一个实用的策略是：**保留最近 N 轮完整的对话**加上一个全局的任务摘要，而不是无限累积所有历史。

### 终止条件（Finish）的设计

在多工具决策的场景中，Agent 不仅要决定"调用什么工具"，还要决定"什么时候停止"。终止条件的设计直接影响 Agent 的行为质量——终止太早，任务可能没有完成；终止太晚，Agent 会做多余的工作甚至陷入无意义的循环。

最常见的终止模式是引入一个专门的 `finish` 工具。当模型判断任务已经完成时，它调用 `finish` 工具（通常带一个 `summary` 参数来总结结果），Agent 循环检测到 `finish` 被调用后退出循环。这种设计的好处是将"是否结束"的判断权交给模型，让它基于上下文自主决定。

但模型的终止判断并不总是准确的。常见的问题包括：**过早终止**（任务只完成了一部分就调用 finish，通常是因为模型"忘记"了完整的任务要求）和**无法终止**（模型反复调用工具但始终不调用 finish，通常是因为模型无法判断结果是否满足要求）。

为了应对这些问题，Agent 通常会设置多层终止机制：第一层是模型自主调用 `finish`（正常终止）；第二层是达到最大轮次限制后强制终止（安全兜底）；第三层是连续 N 轮没有新进展时终止（死循环检测）。这种分层设计确保 Agent 在任何情况下都不会永远运行下去。

终止时的输出设计也值得关注。Agent 不应该在 `finish` 时只返回最后一个工具的结果，而应该根据整个任务过程汇总出一个对用户有意义的最终答案。这个汇总可以由模型在调用 `finish` 时完成，也可以在循环结束后用一次额外的 LLM 调用来生成。

## 新手最容易卡住的点

- **工具描述写得太相似**：多个工具的 description 过于笼统，导致模型频繁选错工具。每个工具的描述应当突出其独特职责。
- **不理解工具链的数据流动**：不清楚前一个工具的返回值是如何传递到下一个工具调用的。关键是理解 Observation 回到上下文后，模型从中提取数据。
- **忘记设计终止条件**：只关注工具的"正向调用"，忽略了何时以及如何结束循环。缺少 `finish` 工具或轮次上限会导致 Agent 无法正常退出。
- **一次给模型太多工具**：工具数量超过模型的有效处理范围后，选择准确率急剧下降。先从少量工具开始，逐步扩展。

## 建议动手实验

- 在 `tool_get_unit_price` 中返回一个更复杂的结果（比如包含多种商品），观察模型是否能正确从中提取并传递信息到 `tool_multiply`。
- 移除 `tool_finish`，观察 Agent 在没有显式终止工具时如何（或无法）结束循环。
- 添加一个新的工具（比如 `tool_discount`），修改任务描述让它在链式调用中出现，测试模型能否正确编排工具顺序。
- 将最大轮次限制设为很小的值（如 2），观察 Agent 在工具链未完成时如何处理被强制终止。
