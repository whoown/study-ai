# 第 10 小时：LangGraph 入门

## 本章定位

用图来表达节点、边和状态流转。上一章我们用 LangChain 的 Agent 实现了动态工具调用，但 Agent 的控制流是隐式的——由框架内部循环驱动，开发者难以精确控制执行顺序和分支逻辑。LangGraph 用显式的图结构解决了这个问题，让 Agent 的编排逻辑变得可视化、可调试、可定制。

## 核心概念

### 图（Graph）作为编排范式的优势

在软件工程中，图是描述"实体与关系"的通用数据结构。将 Agent 的工作流建模为图意味着：每个处理步骤是一个节点，步骤之间的流转关系是一条边。这种建模方式有几个显著优势。

第一是**可视化**。图结构可以直接绘制成流程图，团队成员能一眼看清 Agent 的完整工作流程——哪些步骤会执行、在什么条件下分支、在哪里循环。这比阅读嵌套的函数调用链直观得多。

第二是**可控性**。在传统的 Agent 循环中，执行路径完全由 LLM 的输出决定，开发者只能事后观察。在图编排中，开发者可以显式定义哪些路径是合法的、在什么条件下走哪条边，从而将 LLM 的决策约束在预定义的拓扑结构内。

第三是**可组合性**。子图可以作为节点嵌入到更大的图中，使得复杂流程可以分层构建和复用。这种组合能力在多 Agent 协作场景中尤为重要。

### Node（节点）：计算单元的抽象

节点是 LangGraph 中最基本的计算单元，每个节点是一个接收状态、执行处理、返回更新后状态的函数。节点的设计遵循函数式编程的理念——输入是当前状态，输出是状态更新，节点本身不应持有副作用以外的内部状态。

在实际使用中，节点通常分为几类：**模型节点**负责调用 LLM 并将结果写入状态，比如本章的 `call_model` 函数；**工具节点**负责执行工具调用并返回结果；**逻辑节点**负责纯粹的条件判断或数据转换，不涉及外部调用。

节点设计的一个重要原则是**单一职责**——每个节点只做一件事。如果一个节点既调用模型又执行工具还做条件判断，那它就应该被拆分成多个节点。单一职责使得节点可以被独立测试、独立替换，也使得图的拓扑结构更清晰。

### Edge（边）：流转控制与条件路由

边定义了节点之间的执行顺序。LangGraph 支持两种边：普通边和条件边。

**普通边**是静态的——从节点 A 到节点 B 的跳转是无条件的，任何时候节点 A 执行完毕都会进入节点 B。适用于线性的处理流程。

**条件边**是动态的——跳转目标由一个路由函数在运行时决定。路由函数接收当前状态，返回下一个节点的标识。这是实现分支逻辑的核心机制。例如，在 ReAct 模式中，模型节点执行后的条件边会检查输出中是否包含工具调用：如果有，跳转到工具节点；如果没有，跳转到结束节点。

条件边的路由函数应当是轻量且确定性的——它根据状态中已有的信息做判断，而不应该在路由过程中执行复杂计算或外部调用。`build_graph` 函数中定义的条件边展示了这种模式的具体实现。

### State（状态）：共享上下文的管理

状态是贯穿整个图执行过程的共享数据结构。每个节点从状态中读取所需的输入，将处理结果写回状态。状态是节点之间传递信息的唯一通道——节点之间不应有直接的函数调用关系。

在 LangGraph 中，状态通常定义为一个 TypedDict 或 Pydantic 模型，包含消息历史、中间计算结果、控制标记等字段。状态的 schema 设计是图编排中最关键的决策之一——它决定了节点之间能交换什么信息、图的功能边界在哪里。

状态设计的常见误区是塞入过多字段。一个好的实践是：只将跨节点共享的数据放入状态，节点内部的临时变量不应暴露到状态中。状态应当是节点间的"最小必要接口"。

### Reducer：状态合并策略

当多个节点对状态的同一字段进行更新时，需要一种策略来决定如何合并这些更新。这就是 Reducer 的作用。

最常见的 Reducer 是消息列表的追加（append）策略——每个节点产生的新消息追加到现有消息列表的末尾，而不是覆盖。在 LangGraph 中，这通常通过 `Annotated[list, operator.add]` 语法来声明。

除了追加，还有替换（replace）策略——新值直接覆盖旧值，适用于状态标记、计数器等场景。更复杂的场景可能需要自定义 Reducer，比如去重合并、优先级合并等。

Reducer 的存在使得多个节点可以安全地并发更新状态，而不会产生竞争条件或数据丢失。这在多 Agent 并行执行的场景中尤为重要。

### LangGraph vs LangChain Agent 的区别

LangChain Agent 和 LangGraph 都能实现动态工具调用的 Agent，但它们的设计哲学和适用场景有本质差异。

LangChain Agent 采用的是**隐式循环**模式：框架内部维护一个"思考 → 行动 → 观察"的循环，开发者通过配置 Agent 类型和工具来间接控制行为。这种模式的优点是使用简单，缺点是当你需要在循环中插入自定义逻辑（比如人工审批、中间状态持久化、条件跳转）时，框架的封装反而成了障碍。

LangGraph 采用的是**显式图**模式：开发者用节点和边构建出完整的执行拓扑，图的每一步转换都是透明可控的。这种模式的学习曲线更陡，但给予了开发者完全的控制权。你可以轻松实现人工介入节点、多分支并行执行、循环次数限制、中间状态检查点等复杂逻辑。

简而言之，如果你的需求是标准的 ReAct 循环，LangChain Agent 足矣；如果你需要自定义执行流程、引入人工审批、支持复杂分支或并行执行，LangGraph 是更合适的选择。实际上，LangChain 团队也在逐步推荐用 LangGraph 替代原有的 Agent 框架。

## 新手最容易卡住的点

- 混淆"状态"和"节点内部变量"。图的状态是节点间共享的全局数据，节点内部的临时变量不会自动同步到状态中。如果某个数据需要在节点间传递，它必须被显式写入状态。
- 条件边的路由函数返回了图中不存在的节点名称，导致运行时报错。路由函数的返回值必须是图中已注册的节点标识。
- 忘记设置图的入口节点（entry point）和结束节点（END），导致图无法启动或无法终止。
- Reducer 选择不当导致消息列表被覆盖而非追加。如果消息历史字段没有配置 append Reducer，每个节点的输出会覆盖之前的消息历史。

## 建议动手实验

- 在 `build_graph` 中新增一个"日志节点"，让它在模型节点和工具节点之间执行，打印当前状态内容，观察图的执行顺序。
- 修改条件边的路由逻辑，增加一个"超过 3 次工具调用则强制结束"的条件，体验如何用图结构实现循环控制。
- 对比同一任务在 LangChain Agent 和 LangGraph 下的实现代码量和可读性差异。
- 故意让路由函数返回一个不存在的节点名称，观察 LangGraph 的报错信息。
