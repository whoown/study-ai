# 第 15 小时：主动检索 Agent

## 本章定位

让 Agent 自己判断何时检索，而非每次都机械地走检索流程。前两章建立了 Embedding 检索和 RAG 管线的基础能力，但那些都是"被动"模式——不论用户问什么，系统都执行检索。本章引入检索决策机制，使 Agent 能在"直接回答"和"先检索再回答"之间做出智能选择。

## 核心概念

### 被动检索 vs 主动检索：为什么不能每次都检索

在标准 RAG 流程中，每个用户问题都会触发一次向量检索。这种"无差别检索"看似安全，实则带来多个问题。

首先是 **效率问题**。很多问题根本不需要外部知识——"你好"、"帮我总结上面的对话"、常识性问题等，检索只会增加延迟和成本。其次是 **噪声引入问题**。对于不需要外部知识的问题，检索到的内容反而会成为噪声，可能误导模型的回答方向。最后是 **成本问题**。每次检索都涉及 Embedding 计算和向量数据库查询，在高并发场景下这些开销不可忽视。

主动检索的核心思想是：Agent 先判断当前问题是否需要外部知识支撑，只在必要时才执行检索。这使得 Agent 从一个"管线中的节点"升级为具备决策能力的"智能体"。

### 检索决策（Decision）的实现方式

检索决策的实现主要有两条路线：基于规则和基于 LLM。

**基于规则的决策** 通过预定义的条件来判断是否检索。典型的规则包括：关键词匹配（问题中包含特定领域术语则检索）、问题类型分类（事实性问题检索、闲聊不检索）、对话上下文分析（新话题检索、追问不检索）等。规则方式的优点是可控、可解释、延迟低；缺点是覆盖面有限、维护成本随规则增多而上升。

**基于 LLM 的决策** 直接让大语言模型判断是否需要检索。通过精心设计的 prompt，让模型分析用户问题并输出"需要检索"或"可以直接回答"的判断。这种方式更灵活，能处理规则难以覆盖的边界情况；但引入了额外的 LLM 调用延迟，且判断质量依赖于 prompt 设计和模型能力。

实践中，两种方式可以组合使用——先用规则快速过滤明确的情况，对规则无法判断的边界情况再交给 LLM 决策。

### 检索时机判断的策略

在具体实现层面，判断"此刻是否需要检索"有多种策略。

**关键词触发** 是最简单的策略。维护一个领域关键词列表，当用户问题中出现这些关键词时触发检索。这种方式适合领域边界清晰的场景，但对自然语言的多样性覆盖不足。

**置信度阈值** 让模型先尝试不检索直接回答，如果模型输出的置信度低于阈值，则触发检索后重新回答。这种方式更智能，但需要模型能可靠地输出置信度估计（实际中 LLM 的自我校准能力有限）。

**LLM 自主判断** 是最灵活的方式——在 prompt 中给模型提供"检索工具"的描述，让模型自行决定是否调用。这与 Function Calling / Tool Use 的思路一脉相承，也是 Agentic RAG 的基础。

### 检索与直接回答的平衡

检索决策本质上是一个"精度-召回"权衡。过于保守（倾向于不检索）会导致该检索时没检索，回答质量下降；过于激进（倾向于检索）则退化为被动检索，失去了主动决策的意义。

找到合适的平衡点需要考虑业务场景。在高准确性要求的场景（如医疗、法律问答），宁可多检索也不能回答错误；在对话体验优先的场景（如闲聊机器人），减少不必要的检索延迟更重要。

此外，还需要为"不确定"的情况设计兜底策略——当决策模块无法确定是否需要检索时，是默认检索（安全优先）还是默认不检索（效率优先），这取决于业务对错误的容忍度。

### Agentic RAG 的概念与趋势

Agentic RAG 是传统 RAG 向 Agent 范式演进的方向。在传统 RAG 中，检索是一个固定的管线步骤；在 Agentic RAG 中，检索变成 Agent 可以主动调用的"工具"之一。

这意味着 Agent 不仅能决定是否检索，还能决定检索什么（改写查询）、从哪里检索（选择知识库）、检索结果是否足够（多轮检索）。Agent 可以在一次回答过程中执行多次检索、对检索结果做验证、甚至在发现信息不足时主动要求用户澄清。

更进一步，Agentic RAG 可以与其他 Agent 能力（如规划、反思、工具调用）结合，构建出能处理复杂多步问题的智能系统。这已经成为当前 AI 应用架构的主流趋势。

### 主动检索的常见问题

**过度检索** 是最常见的问题——决策逻辑过于保守，几乎每个问题都会触发检索，主动检索退化为被动检索。通常是因为决策规则或 prompt 设计不够精确。

**检索不足** 与之相反——Agent 在该检索时选择直接回答，导致输出了基于参数知识的不准确回答。这在用户提出特定领域问题但措辞日常化时尤为常见。

**延迟增加** 是引入 LLM 决策层的代价。基于 LLM 的决策本身需要一次模型调用，如果判断为"需要检索"，整个流程比直接检索多了一步。可以通过优化决策 prompt（减少 token）、使用更小的决策模型、或缓存常见决策结果来缓解。

## 新手最容易卡住的点

- 把 `rule_based_decide` 和 `llm_decide` 看作互斥方案——实际上它们可以级联使用，规则先行、LLM 兜底。
- 忽略了"不检索时回答质量可能下降"的风险——主动检索的价值在于智能决策，而非简单地减少检索次数。
- 在测试时只用明显需要/不需要检索的案例，忽略了边界情况（如问题看似闲聊但实际需要领域知识）。
- 混淆了"检索决策"和"检索执行"——决策是判断是否需要检索，执行是具体的向量检索过程，两者是独立的模块。

## 建议动手实验

- 准备一组混合问题（事实性问题、闲聊、常识问题），分别用 `rule_based_decide` 和 `llm_decide` 测试决策结果，对比两者的判断差异。
- 修改 `rule_based_decide` 的规则条件，观察决策边界的变化。
- 在 `run_active_agent` 中故意跳过检索步骤（始终直接回答），对比回答质量的变化。
- 设计一个需要多轮检索才能完整回答的问题，观察当前单轮检索架构的局限性。
