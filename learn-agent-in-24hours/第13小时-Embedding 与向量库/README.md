# 第 13 小时：Embedding 与向量库

## 本章定位

建立语义向量与相似度检索的直觉。本章从最基础的"文本如何变成数字"出发，介绍 Embedding 的本质、向量空间中的语义关系，以及如何利用向量数据库高效地进行相似度检索。掌握这些概念是后续 RAG、主动检索等高级功能的前提。

## 核心概念

### Embedding 的本质：从离散符号到连续向量空间

在传统的计算机表示中，文本是离散的符号序列——每个词只是一个独立的 token ID，词与词之间没有数学上的距离关系。"猫"和"狗"在 one-hot 编码下的距离，与"猫"和"经济学"的距离完全相同，这显然不符合人类的语义认知。

Embedding 的核心目标是将离散符号映射到一个连续的高维向量空间中，使得语义相近的文本在这个空间中的距离也相近。这种映射由神经网络学习得到：模型在大规模语料上训练，逐步调整每个 token 对应的向量位置，直到向量间的几何关系能够反映语义关系。

Embedding 并非只针对单个词。现代 Embedding 模型（如 OpenAI 的 text-embedding 系列或开源的 sentence-transformers）可以将整个句子甚至段落编码为一个固定维度的向量。这意味着我们可以用一个向量来"概括"一段文本的含义，并通过向量运算来度量文本之间的语义相似程度。

### 向量空间中的语义关系

当文本被映射为向量后，语义关系就变成了几何关系。"机器学习"和"深度学习"的向量会在空间中彼此靠近，而"机器学习"和"法国料理"则相距甚远。这种性质不仅体现在同义词层面，还能捕捉更复杂的关系——比如经典的 word2vec 实验中，"国王 - 男人 + 女人 ≈ 女王"。

这种几何化的语义表示是向量检索得以工作的根基。当用户提出一个问题时，我们将问题编码为向量，然后在已有的文档向量集合中寻找距离最近的那些——它们大概率就是与问题语义最相关的文档。

需要注意的是，向量空间中的语义关系并非完美。同一个词在不同上下文中可能有完全不同的含义（多义词问题），而 Embedding 模型的训练数据和目标函数也会引入特定的偏差。理解这些局限性对于设计可靠的检索系统至关重要。

### 相似度度量方式：余弦相似度、欧式距离、点积

有了向量表示后，需要一种数学方法来衡量"两个向量有多相似"。常见的度量方式有三种。

**余弦相似度（Cosine Similarity）** 计算两个向量夹角的余弦值，取值范围为 [-1, 1]。它只关注向量的方向而忽略长度，因此对文本长度不敏感。在大多数文本语义检索场景中，余弦相似度是默认选择。

**欧式距离（L2 Distance）** 计算两个向量在空间中的直线距离。它同时受方向和长度影响。在归一化向量上，欧式距离与余弦相似度有单调对应关系，因此两者的排序结果一致。

**点积（Dot Product）** 是余弦相似度的"未归一化版本"，同时考虑方向相似性和向量长度。在某些场景下（如向量长度编码了文本重要性），点积可能比余弦相似度更合适。

选择哪种度量方式取决于具体场景和所用 Embedding 模型的训练目标。多数 Embedding 模型默认使用余弦相似度进行评测，因此在没有特殊理由时，余弦相似度通常是最稳妥的选择。

### 向量归一化（Normalize）的作用

向量归一化是指将向量缩放为单位长度（L2 范数为 1），数学上即 v / ||v||。归一化后的向量只保留方向信息，去掉了长度差异。

归一化的实际意义在于：对归一化向量做点积就等价于余弦相似度，这大幅简化了计算。许多向量数据库内部会在索引构建时自动归一化，但如果你自己实现检索逻辑，显式归一化是保证结果正确性的必要步骤。

此外，归一化还能消除因文本长度不同导致的向量模长差异，使得检索结果更公平地反映语义相关性而非文本长度。

### 向量数据库（Vector Store）的种类与选型

向量数据库专门为高维向量的存储和近似最近邻（ANN）检索而设计。主流方案各有特点。

**Chroma** 是一个轻量级、嵌入式的向量数据库，适合本地开发和中小规模场景。它的 API 简洁，与 Python 生态集成良好，非常适合快速原型验证。

**FAISS**（Facebook AI Similarity Search）是 Meta 开源的高性能向量检索库。它提供多种索引类型（Flat、IVF、HNSW 等），支持 GPU 加速，适合对性能有较高要求的场景。但 FAISS 本身只是检索引擎，不包含数据管理功能。

**Pinecone** 是全托管的云端向量数据库服务，提供自动扩缩容、多租户隔离等企业级特性。适合不想自行运维的生产部署。

**Milvus** 是一个分布式向量数据库，支持十亿级向量的存储与检索，提供丰富的索引类型和混合查询能力。适合大规模生产环境。

选型时的核心考量包括：数据规模（百万级以下还是十亿级）、部署形态（嵌入式、单机还是分布式）、是否需要元数据过滤、延迟要求、以及团队的运维能力。本章使用 Chroma 作为教学示例，因为它零配置、开箱即用。

### Embedding 模型的选择

Embedding 模型的质量直接决定了检索效果的上限。目前主要有两条路线。

**商业模型** 以 OpenAI 的 text-embedding-3-small/large 为代表，优势是效果好、维度灵活、接入简单，但存在 API 调用成本和数据隐私顾虑。

**开源模型** 以 sentence-transformers 系列（如 all-MiniLM-L6-v2、bge-large-zh 等）为代表，优势是可本地部署、无 API 费用、数据不出域。对于中文场景，选择专门针对中文优化的模型（如 bge 系列、m3e 系列）通常效果更好。

选择模型时需要考虑：目标语言、向量维度（影响存储成本和检索速度）、模型大小（影响推理延迟）、以及在目标领域上的检索准确率。MTEB 排行榜是评估 Embedding 模型效果的重要参考。

### 常见问题与优化方向

**维度灾难** 是高维向量空间中的固有问题——随着维度增加，数据点之间的距离趋于均匀，区分度下降。实践中通常选择 384-1536 维的 Embedding，在语义表达力和计算效率之间取得平衡。

**语义漂移** 指 Embedding 模型的训练语料与实际使用场景之间的领域差异，导致向量无法准确反映目标领域的语义关系。解决方案包括在领域数据上微调 Embedding 模型，或使用 prompt 引导模型生成更契合目标领域的向量。

**冷启动问题** 出现在知识库初建阶段——文档量少、覆盖面窄，导致检索经常找不到相关结果。可以通过设置相似度阈值来识别"检索不到"的情况，并提供兜底策略而非返回低质量结果。

## 新手最容易卡住的点

- 混淆 Embedding 维度和语义质量的关系——维度越高不一定越好，关键在于模型的训练质量和数据分布。
- 直接比较未归一化的向量点积，导致结果被向量长度主导而非语义方向。
- 用简单的哈希函数理解 Embedding 的直觉是好的，但不要忘记真实 Embedding 模型的向量包含了上下文语义信息，远比哈希丰富。
- 忽略相似度阈值的设置——检索到的"最近邻"不代表真的相关，需要设定一个最低相似度阈值来过滤噪声结果。

## 建议动手实验

- 用 `hash_embedding` 对几组语义相近和语义无关的文本生成向量，观察 `cosine_similarity` 的差异，思考简化向量函数的局限性。
- 在 `run_chroma_demo` 中插入一些与已有文档语义接近但措辞不同的查询，验证向量检索是否能跨越措辞差异找到相关内容。
- 尝试不调用 `l2_normalize` 就直接计算点积，对比归一化前后的排序结果是否一致。
- 修改向量维度（调整 `hash_embedding` 的输出长度），观察对检索精度的影响。
