# 第 8 小时：上下文管理

## 本章定位

在有限的上下文窗口里做信息取舍。上一章实现了短期记忆，但消息列表会无限增长，最终撞上模型的 token 上限。本章引入上下文管理策略，学习如何在有限空间内保留最有价值的信息，使 Agent 能支撑更长的会话。

## 核心概念

### Context Window 的物理限制

Context Window（上下文窗口）是大语言模型在单次推理中能处理的最大 token 数量，它是由模型架构在训练阶段决定的硬性约束。不同模型的窗口大小差异显著：早期 GPT-3.5 为 4K token，GPT-4 最高支持 128K，Claude 系列则达到 200K。

理解这个限制的关键在于：上下文窗口是输入和输出共享的。如果你的消息历史占用了 120K token，而窗口上限是 128K，那么模型最多只能生成约 8K token 的回复。此外，即便在窗口范围内，模型对超长上下文中信息的利用效率也会下降——研究表明，信息在上下文的开头和结尾被关注的概率更高，中间部分容易被"遗忘"，这被称为"Lost in the Middle"现象。

因此，上下文管理并非简单的"塞不下就截断"，而是一门在有限空间内最大化信息价值的工程学科。

### Token 计算与预算管理

Token 是 LLM 处理文本的基本单位，但 token 与字符并非一一对应。英文中一个 token 大约对应 4 个字符或 0.75 个单词；中文中一个汉字通常对应 1-2 个 token。工具调用的 JSON 结构、函数签名描述等也会占用 token，且这部分开销容易被低估。

在实际工程中，需要建立明确的 token 预算体系：系统提示词占用多少 token？工具描述占用多少？为用户输入预留多少？为模型回复预留多少？一个常见的分配策略是：系统提示 10-15%，工具描述 10-20%，历史消息 40-50%，当前输入 10%，回复预留 15-20%。

预算管理的核心函数通常会在每次调用模型前计算当前消息列表的 token 总量，判断是否超出预算，如果超出则触发截断或压缩策略。`build_system_with_memory` 正是在这个环节发挥作用的。

### 截断策略（Truncation）

当消息历史超出预算时，截断是最直接的应对方式。常见的截断策略包括：

**FIFO（先进先出）**：删除最早的消息，保留最近的消息。这是最简单的策略，假设越近的信息越重要。它的问题在于可能丢失对话开头的关键上下文——比如用户最初定义的任务目标。

**滑动窗口**：保留固定数量的最近 N 轮对话。比 FIFO 更可控，但本质逻辑相同——优先保留近期消息。滑动窗口通常会固定保留 system 消息，只对 user/assistant/tool 消息做窗口滑动。

**优先级截断**：为不同类型的消息赋予不同的保留优先级。例如，system 消息和包含关键决策的消息优先级最高，闲聊和重复确认优先级最低。实现上需要一个评分机制来判断消息的重要性，复杂度显著高于前两种策略。

选择哪种截断策略取决于应用场景：任务型对话适合优先级截断，闲聊场景用滑动窗口即可，快速原型开发通常从 FIFO 开始。

### 摘要压缩（Summary）

摘要压缩是比截断更精细的策略——不是丢弃消息，而是用 LLM 将多条消息压缩成一段简短的摘要，然后用摘要替代原始消息。

典型的实现流程是：当消息列表超过阈值时，取出最早的 N 条消息，调用 LLM 生成一段摘要文本，然后将这 N 条消息替换为一条 system 或 user 角色的摘要消息。这样既保留了关键信息，又大幅减少了 token 占用。

摘要压缩的挑战在于：首先，生成摘要本身需要一次额外的 LLM 调用，增加了延迟和成本；其次，摘要不可避免地会丢失细节，如果后续对话需要引用被摘要掉的具体细节，Agent 就无法准确回答；最后，摘要的质量直接影响后续对话的质量，一个糟糕的摘要可能比直接截断更有害。

实践中，摘要压缩常与滑动窗口结合使用：近期消息保持原文，较早的消息做摘要，形成"摘要 + 近期原文"的混合结构。

### 上下文预算分配策略

一个成熟的上下文管理系统需要明确的预算分配策略。以 128K token 窗口为例，一种典型的分配方案如下：

**固定区域**：系统提示词（约 2K）+ 工具描述（约 3K）+ 回复预留（约 4K），合计约 9K，这部分不可压缩。

**弹性区域**：剩余约 119K 分配给历史消息和当前输入。当历史消息过多时，触发压缩或截断。

预算分配需要考虑动态调整：如果当前任务涉及大量工具调用，工具描述的开销会增大，历史消息的可用空间就需要相应压缩。`run_session` 函数在每轮对话中协调这些分配决策。

### 上下文管理的工程挑战与最佳实践

**Token 计算的准确性**：不同模型使用不同的 tokenizer，同一段文本在不同模型上的 token 数可能不同。工程实践中应使用对应模型的官方 tokenizer 库（如 tiktoken）进行精确计算，而不是用经验公式估算。

**系统提示的不可压缩性**：系统提示是 Agent 行为的锚点，在任何截断策略中都应被保护。一旦系统提示被部分截断，Agent 的行为可能发生不可预期的偏移。

**工具消息的特殊处理**：tool_call 和 tool_result 消息必须成对存在。如果截断时只删除了其中一条，会导致消息列表结构破损，模型可能报错或产生混乱输出。因此截断逻辑需要以"消息组"而非"单条消息"为单位操作。

**监控与告警**：生产环境中应监控每次调用的 token 使用量、截断/压缩触发频率、摘要质量评分等指标，在问题发生前主动预警。

## 新手最容易卡住的点

- token 计算不准确导致的意外截断或超限。不要用"一个汉字约等于一个 token"这种粗略估算做预算管理，应使用 tiktoken 等库进行精确计算。
- 截断时破坏了 tool_call/tool_result 的配对关系，导致模型报错。截断逻辑应以完整的"消息组"为单位操作。
- 摘要质量不佳导致后续对话出现"幻觉"。如果摘要中遗漏了某个关键事实，模型可能会自行编造信息来填补空白。
- 忘记为模型回复预留 token 空间。如果整个窗口都被输入占满，模型没有空间生成回复，会导致输出被截断或为空。

## 建议动手实验

- 在 `build_system_with_memory` 中添加 token 计数日志，观察系统提示、历史消息和工具描述各自占用的 token 比例。
- 实现一个简单的 FIFO 截断策略：当消息超过 10 条时，删除最早的非 system 消息，观察 Agent 是否"忘记"了早期的关键信息。
- 在长对话中让 Agent 引用第一轮提到的信息，对比截断前后的回答差异。
- 尝试用 LLM 对前 5 轮对话生成摘要，替换原始消息，观察对后续对话质量的影响。
