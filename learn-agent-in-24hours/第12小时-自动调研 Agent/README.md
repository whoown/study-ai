# 第 12 小时：自动调研 Agent

## 本章定位

把搜索、筛选和总结串成小型实战流程。前面十一章积累的所有能力——工具调用、记忆、上下文管理、框架使用——在本章汇聚为一个完整的应用场景。自动调研 Agent 是 LLM 应用中最典型的落地形态之一，它体现了 Agent 从"能对话"到"能做事"的质变。

## 核心概念

### 调研 Agent 的典型 Pipeline 设计

一个调研 Agent 的工作流程通常由四个阶段组成：**理解需求 → 搜索信息 → 筛选评估 → 综合总结**。这四个阶段形成一条串行的处理管线（pipeline），每个阶段的输出是下一个阶段的输入。

**需求理解阶段**：Agent 接收用户的调研问题，通过 LLM 将模糊的自然语言需求转化为具体的搜索策略——确定关键词、明确搜索范围、拆分子问题。这一步的质量直接决定了后续搜索的方向是否正确。

**搜索阶段**：根据搜索策略调用搜索工具（如 `mock_search`）获取候选信息源。实际场景中可能涉及多个搜索引擎、数据库、API 的并行查询。

**筛选评估阶段**：对搜索结果进行质量评估和相关性排序，过滤掉低质量或不相关的内容。`_score_doc` 和 `list_sources` 就是这一阶段的核心函数。

**综合总结阶段**：将筛选后的高质量信息源交给 LLM，生成结构化的调研报告。这一步需要 LLM 具备信息整合、去重、对比和归纳的能力。

Pipeline 的设计关键在于各阶段之间的接口定义——每个阶段输出什么格式的数据？下一阶段期望接收什么输入？接口定义越清晰，pipeline 的可维护性和可测试性就越好。

### 信息源管理与可信度评估

调研的质量很大程度上取决于信息源的质量。信息源管理包括两个层面：**来源多样性**和**可信度评估**。

来源多样性是指尽量从不同类型的信息源获取信息——学术论文、技术博客、官方文档、社区讨论等。单一来源容易导致信息偏差，多源交叉验证能显著提高调研结论的可靠性。

可信度评估是对每个信息源的质量进行量化判断。常见的评估维度包括：来源权威性（官方文档 > 个人博客）、时效性（近期发布 > 多年前的文章）、内容完整度（详细分析 > 简短提及）、与查询的相关度（直接回答问题 > 间接相关）。

在实际实现中，可信度评估可以是基于规则的（如本章的 `_score_doc` 函数），也可以是基于 LLM 的——让 LLM 阅读每个信息源的摘要后给出质量评分。前者速度快但灵活性低，后者灵活但成本高。混合方案是常见的工程选择：先用规则做粗筛，再用 LLM 做精排。

### 评分（Scoring）策略

评分是将定性判断转化为定量指标的过程，它直接决定了哪些信息会进入最终的总结阶段。

**加权评分**是最常见的策略：为每个评估维度分配一个权重，计算加权总分。例如相关性权重 0.4、权威性 0.3、时效性 0.2、完整度 0.1。权重的设定通常需要根据具体的调研场景调整——技术选型调研可能更看重时效性，学术综述更看重权威性。

**阈值过滤**：设定一个最低分数阈值，低于阈值的信息源直接淘汰。阈值的设定需要平衡精度和召回——阈值太高可能过滤掉有价值的信息，太低则引入过多噪声。

**Top-K 选取**：按分数排序后取前 K 个结果。这种策略保证了进入总结阶段的信息源数量可控，适合对输出长度有明确要求的场景。

评分策略的一个重要原则是**可解释性**。开发者和用户都应该能理解为什么某个信息源得分高/低。`_score_doc` 的实现应当清晰地展示评分逻辑，方便后续调试和优化。

### Pipeline 编排：串行 vs 并行

调研 pipeline 的各阶段之间存在数据依赖，但并非所有步骤都必须串行执行。

**串行执行**是最简单的编排方式：搜索完成后才开始筛选，筛选完成后才开始总结。实现简单、调试方便，但效率较低——如果有多个搜索源需要查询，串行执行会使总耗时等于各搜索源耗时之和。

**并行执行**可以在有独立性的步骤之间引入。典型的并行机会包括：多个搜索源的并行查询（搜索 Google 和搜索 Arxiv 可以同时进行）、多个文档的并行评分（对文档 A 和文档 B 的评分是独立的）。并行执行能显著降低总耗时，但增加了错误处理和结果合并的复杂度。

在本章中，`build_agent` 将搜索和评分工具装配到 Agent 中，由 LLM 决定调用顺序。`run_with_llm` 和 `run_offline_pipeline` 分别展示了在线（LLM 驱动）和离线（规则驱动）两种编排模式。在线模式更灵活，离线模式更可控——选择哪种取决于你对调研流程的确定性要求。

### 调研 Agent 的质量保障与常见陷阱

**幻觉问题**：LLM 在总结阶段可能编造信息源中不存在的内容。应对策略包括：在提示词中明确要求"仅基于提供的信息源进行总结"、要求 Agent 为每个结论标注出处、对生成的总结进行事实核查。

**信息过时**：搜索结果中可能包含过时的信息，如果不加时效性筛选，调研报告的结论可能基于已过时的数据。评分策略中应包含对发布时间的考量。

**搜索偏差**：搜索关键词的选择直接影响结果的覆盖面。如果关键词过于狭窄，可能遗漏重要信息；过于宽泛则引入大量噪声。好的实践是让 LLM 生成多组关键词从不同角度搜索，然后合并去重。

**总结的深度与广度平衡**：用户期望的调研报告可能是"全面概览"也可能是"深度分析"。Agent 需要根据用户需求调整总结的粒度。在提示词设计中明确输出格式和详细程度的要求能有效控制这一点。

**成本控制**：一次调研可能涉及多次 LLM 调用（需求理解、评分、总结），加上搜索 API 的调用费用，成本可能快速累积。在 pipeline 设计中应考虑成本上限和降级策略——当成本接近预算时，减少搜索轮次或降低评分精度。

## 新手最容易卡住的点

- 搜索结果为空或全部低质量时 Agent 不知道如何处理。pipeline 应当为"无有效结果"场景设计降级路径——比如扩大搜索范围、放宽评分阈值、或向用户反馈"未找到足够信息"。
- 评分逻辑中硬编码了特定领域的假设（如"论文一定比博客可信"），在切换领域时产生错误的排序结果。评分维度的权重应该是可配置的。
- 将搜索结果全量传入总结阶段，导致上下文窗口溢出。筛选阶段的 Top-K 策略不仅是为了质量，也是为了控制输入规模。
- 忽视 `run_offline_pipeline` 的价值。离线 pipeline 不依赖 LLM API，是调试搜索和评分逻辑的理想工具。

## 建议动手实验

- 修改 `_score_doc` 的评分权重，观察排序结果的变化，体会不同权重设定对最终调研质量的影响。
- 在 `mock_search` 中添加一些明显低质量的结果（如内容为空、标题与查询完全无关），验证评分系统能否将其过滤掉。
- 先运行 `run_offline_pipeline` 确认搜索和评分逻辑正确，再运行 `run_with_llm` 对比 LLM 驱动的调研效果。
- 尝试修改总结阶段的提示词，要求 Agent 为每个结论标注信息来源，观察输出格式的变化。
