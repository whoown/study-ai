# 第 24 小时：部署上线

## 本章定位

把 Agent 包装成可访问服务。前 23 章构建的 Agent 都是"在本地运行一次就结束"的脚本形态。要让 Agent 真正可用，必须将它封装为可被远程调用的服务——对外暴露 API 端点，对内管理 Agent 实例的生命周期、资源和安全。本章围绕应用工厂模式和双模式设计，展示 Agent 服务化的基本框架。

## 核心概念

### Agent 服务化的基本架构

Agent 服务化的本质是将"一次性脚本"转变为"持续运行的服务进程"。一个最基本的 Agent 服务包含三层：接入层（接收外部请求、返回响应）、业务层（Agent 的推理和执行逻辑）、以及依赖层（LLM API、数据库、外部工具等）。

接入层通常是一个 Web 框架（如 FastAPI、Flask），它负责 HTTP 协议的处理、请求路由、参数校验和响应序列化。业务层是 Agent 本身，与前面章节中的逻辑一致。依赖层是 Agent 运行所需的外部资源。

服务化带来的关键变化是"并发"和"状态管理"。脚本形态的 Agent 一次只处理一个请求；服务形态的 Agent 需要同时处理多个请求，且不同请求之间的状态不能互相干扰。这意味着 Agent 的设计必须考虑线程安全或采用异步架构，共享资源（如模型连接池）需要妥善管理。

### API 设计：RESTful vs WebSocket vs gRPC

Agent 服务对外暴露的 API 接口设计，直接影响客户端的集成体验和系统的性能特性。

RESTful API 是最通用的选择：客户端发送 HTTP 请求，服务端返回完整的响应。它简单、无状态、易于调试和缓存。但对于 Agent 场景，RESTful 有一个明显的问题——Agent 的推理可能需要数秒甚至数十秒，客户端在等待期间没有任何反馈。

WebSocket 通过建立持久的双向连接解决了这个问题。Agent 可以在推理过程中逐步推送中间状态（如"正在分析…""正在搜索…""正在生成…"），客户端实时展示进度。这对用户体验有巨大提升，但服务端需要管理长连接的生命周期。

gRPC 提供了高性能的二进制通信和流式传输能力，适合服务间调用（如 Agent 服务调用 Tool 服务）。它的 streaming RPC 特性也能支持类似 WebSocket 的逐步推送。但 gRPC 对浏览器客户端的支持不如 HTTP 原生，通常需要 gRPC-Web 中间层。

实践中的常见方案是"对外 RESTful + SSE（Server-Sent Events）、对内 gRPC"。SSE 是 RESTful 的单向流式扩展，允许服务端向客户端推送事件流，实现上比 WebSocket 简单且兼容性更好。

### 应用工厂（App Factory）模式

本章的 `create_app` 函数采用了应用工厂模式——不在模块级别创建应用实例，而是通过一个工厂函数按需创建。调用者可以传入不同的配置参数，获得不同配置的应用实例。

应用工厂模式的核心价值体现在三个方面。第一是可测试性：测试时可以创建使用 mock 配置的应用实例，不会影响生产配置。第二是多实例支持：同一个进程中可以创建多个不同配置的应用实例（如一个使用真实 LLM、另一个使用 mock）。第三是配置延迟绑定：应用的配置在创建时才确定，而不是在导入模块时就固定，这使得通过环境变量或配置文件控制行为成为可能。

与之相对的是"单例模式"——在模块级别创建唯一的全局应用实例。单例模式在简单场景下更直接，但在需要多配置或可测试性时会遇到困难。应用工厂模式在 Flask、FastAPI 等框架的最佳实践中被广泛推荐。

### Mock/Real 双模式的工程价值

本章同时实现了 `_answer_with_llm`（真实 LLM 调用）和 `_answer_mock`（模拟响应）两种模式。这种双模式设计是贯穿整个课程的工程模式，在服务化场景中其价值更加突出。

在开发阶段，mock 模式让你可以快速迭代 API 接口、请求格式和响应结构，而不需要消耗 LLM API 配额。在测试阶段，mock 模式提供确定性的输出，使得自动化测试可以可靠地验证端到端行为。在生产环境，当 LLM 服务出现故障时，mock 模式可以作为降级方案提供基本的响应能力。

双模式的切换通常通过环境变量或配置参数控制。在应用工厂模式中，这非常自然——创建应用时传入不同的参数即可切换模式。好的实践是确保两种模式的接口签名完全一致，使得切换对调用方透明。

### 部署方案：容器化、Serverless、边缘部署

Agent 服务的部署方案选择取决于流量模式、延迟要求、成本预算和运维能力。

容器化部署（Docker + Kubernetes）是最主流的方案。将 Agent 服务打包为 Docker 镜像，通过 Kubernetes 管理部署、扩缩容和健康检查。优势是环境一致性强、扩展灵活、生态成熟。劣势是运维复杂度较高，对于小团队可能过于重量级。

Serverless 部署（如 AWS Lambda、Google Cloud Functions）适合流量波动大的场景。服务按请求计费，空闲时不产生成本。但 Serverless 有冷启动延迟的问题，且单次执行的时间和资源受到严格限制（通常 15 分钟以内），对于推理时间长的 Agent 可能不够。

边缘部署是新兴的方向——将 Agent 服务部署到离用户最近的边缘节点（如 Cloudflare Workers、Deno Deploy），最大程度降低网络延迟。但边缘环境的计算资源有限，通常只能运行轻量级的推理逻辑，复杂的 Agent 仍需回源到中心服务。

对于本课程的场景，建议从最简单的方案开始：单机 Docker 部署或直接用 Python 进程运行，满足需求后再考虑更复杂的方案。

### 生产环境的关键考量

将 Agent 服务推向生产环境时，有几个必须解决的问题。

限流（Rate Limiting）是保护服务的第一道防线。无论是防止恶意攻击还是避免意外的流量峰值压垮服务，都需要对 API 端点设置请求频率限制。常见的限流策略包括固定窗口、滑动窗口和令牌桶。

监控（Monitoring）覆盖三个维度：基础设施监控（CPU、内存、网络）、应用监控（请求量、延迟、错误率）和业务监控（Agent 完成率、回答质量评分）。至少应配置基本的健康检查端点和关键指标的告警。

日志（Logging）在 Agent 服务中尤为重要，因为 LLM 的输出不确定性意味着问题往往难以复现。结构化日志（JSON 格式）配合请求级别的 trace ID，是排查问题的基础。

成本控制是 Agent 服务特有的挑战。每次 Agent 执行都涉及 LLM API 调用费用，如果不加控制，一次恶意或异常的请求可能消耗大量预算。应在服务层面设置单次请求的 token 上限和单用户的日消费上限。

### Agent 服务的安全性

Agent 服务的安全性需要从认证、授权和输入过滤三个层面考虑。

认证（Authentication）确认"你是谁"——通过 API Key、JWT Token 或 OAuth2 等机制验证调用方身份。对于面向公众的服务，认证是必须的。

授权（Authorization）确认"你能做什么"——不同的调用方可能有不同的权限级别。例如普通用户只能使用基本的问答功能，管理员可以访问调试接口。

输入过滤是 Agent 服务特有的安全需求。因为用户输入会直接进入 LLM 的提示词，恶意的 prompt injection 攻击可能绕过 Agent 的安全限制。防御手段包括：输入长度限制、敏感关键词过滤、指令注入检测（专门的分类模型判断输入是否包含注入攻击），以及输出审核（在返回给用户前检查 Agent 输出是否包含不当内容）。

### 可扩展性设计

当请求量增长到单机无法承受时，你需要考虑水平扩展的策略。

无状态设计是水平扩展的前提。如果 Agent 服务的每次请求处理都不依赖本地状态（会话状态存储在 Redis 等外部存储中），那么可以简单地增加服务实例来分摊负载。

异步任务队列适合处理时间长的 Agent 请求。客户端提交请求后立即获得一个任务 ID，Agent 在后台处理，客户端通过轮询或 WebSocket 获取结果。这种方式释放了 Web 服务器的连接资源，也便于实现优先级排队和负载均衡。

缓存是最容易被忽视但效果显著的优化。对于相同或相似的输入，Agent 的输出往往也是相同的。通过对输入做语义哈希并缓存结果，可以大幅减少重复的 LLM 调用，同时降低延迟和成本。

## 新手最容易卡住的点

- 将 Agent 服务简单等同于"加个 Flask 路由"——服务化涉及并发处理、错误恢复、安全防护等一系列问题，单纯暴露 HTTP 端点只是第一步。
- 忽视 Mock 模式在服务化中的价值——部署和调试 API 接口时，频繁调用真实 LLM 既昂贵又缓慢。先用 mock 模式验证服务框架的正确性，再切换到真实模式。
- 不理解应用工厂模式的目的——如果你发现自己在代码中到处硬编码配置参数，说明你还没有理解工厂模式带来的配置灵活性。

## 建议动手实验

- 在 `create_app` 创建的应用中新增一个健康检查端点（`/health`），返回服务状态和版本信息，这是生产部署的基本要求。
- 通过环境变量控制 mock/real 模式切换，体会应用工厂模式如何实现配置外部化。
- 使用并发工具（如 `ab` 或 `wrk`）对服务发送多个并行请求，观察服务在并发场景下的行为。
- 为 API 端点添加一个简单的 API Key 认证中间件，体会安全层如何与业务逻辑解耦。
- 编写一个 Dockerfile 将服务打包为容器镜像，完成从代码到可部署产物的全流程。
