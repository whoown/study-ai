# 第 22 小时：多智能体实战

## 本章定位

把角色、技能和状态合成完整系统。前面几章分别学习了 Skills 系统、多智能体分工和编排模式，本章将这些概念整合到一个端到端的实战项目中——包含技能注册表、多角色节点（PM、Dev、QA、Publisher）、Supervisor 调度和完整的图编排，模拟一个真实的团队协作流程。

## 核心概念

### 完整 Workflow 的设计方法论

设计一个完整的多智能体 Workflow，需要回答四个层次的问题。第一层是"目标分解"：最终交付物是什么？需要哪些中间产出物？每个产出物由谁负责？第二层是"流程编排"：这些角色按什么顺序协作？哪些步骤可以并行？哪些步骤需要迭代？第三层是"数据流设计"：每个角色需要什么输入？产出什么输出？上下文如何在角色间流转？第四层是"异常处理"：每个环节可能出什么错？如何降级或重试？

本章的实战项目覆盖了这四个层次。PM 负责需求拆解，Dev 负责技术实现（调用技能），QA 负责质量验证，Publisher 负责最终发布，Supervisor 负责全局调度。这个角色分配不是随意的——它映射了真实软件团队的分工模式，使得系统行为符合人类对协作流程的直觉理解。

设计 Workflow 时的一个关键原则是"先跑通再优化"。不要一开始就追求完美的角色划分和流程设计。先用最少的角色跑通端到端流程，再根据实际运行中发现的问题逐步增加角色、调整流程。本章的多角色设计是经过这个演进过程后的结果。

### 契约（Contract）：Agent 间的输入输出约定

在多角色系统中，角色之间的交互必须遵循明确的契约。契约定义了每个角色的输入数据格式、输出数据格式、以及调用前提条件。没有契约的多智能体系统就像没有 API 文档的微服务——能跑但不可维护。

契约的形式可以是简单的文档约定，也可以是强类型的 schema 定义。前者灵活但容易出错，后者严格但修改成本高。实践中的建议是：在原型阶段使用松散的字典约定（快速迭代），在系统稳定后逐步迁移到强类型定义（如 Pydantic 模型或 JSON Schema）。

契约违反是多智能体系统中最常见的 bug 来源。典型场景是：Dev 角色的输出格式发生了变化（如从字符串变为列表），但 QA 角色仍按旧格式解析。解决方案是在交接点增加验证逻辑——每个角色在处理输入前先校验格式，在产出输出后也进行自检。

### 可观测性（Observability）：日志、Trace、指标

可观测性是将多智能体系统从"能跑的 demo"升级为"可维护的产品"的关键能力。可观测性包含三个支柱：日志（Logs）记录离散事件，追踪（Traces）记录请求的完整生命周期，指标（Metrics）记录系统级别的聚合数据。

在多智能体系统中，可观测性的挑战在于请求会跨越多个 Agent。一个用户请求可能经过 Supervisor → PM → Dev → QA → Publisher 的链路，每一步都有 LLM 调用和状态变更。你需要一个贯穿整条链路的 trace ID，使得可以从任何一个节点追溯到完整的执行路径。

实际落地时，最低成本的做法是：在每个角色节点的入口和出口打印结构化日志（包含 trace ID、角色名、输入摘要、输出摘要、耗时），这足以支撑大部分调试需求。更进一步可以接入 LangSmith、Phoenix 等 LLM 可观测性平台，获得自动化的 trace 可视化和成本统计。

### 团队（Team）协作模式

多智能体的协作模式可以分为三大类：串行、并行和混合。

串行模式是最简单的——角色按固定顺序依次执行。它的优势是确定性强、实现简单，但总耗时是所有角色耗时之和，且不能利用任务的并行性。

并行模式让多个角色同时工作。例如 PM 拆解出三个子任务后，三个 Dev 实例可以同时处理。并行模式大幅缩短了总耗时，但引入了同步和合并的问题——三个 Dev 的产出如何合并成一个整体？如果产出之间有冲突怎么办？

混合模式结合了串行和并行的优势。典型的模式是"阶段内并行、阶段间串行"：在 Dev 阶段多个实例并行工作，在 QA 阶段统一验证所有产出，在 Publisher 阶段串行执行最终发布。本章的实现更接近串行模式，但理解并行和混合模式对于生产环境的设计至关重要。

### 从原型到生产的工程化要点

原型阶段的代码关注"能不能跑通"，生产环境的代码关注"能不能稳定运行、出了问题能不能快速恢复"。两者之间存在巨大的鸿沟。

生产化的关键要点包括：错误重试机制（LLM API 调用可能因限流或网络问题失败）、超时控制（单个角色的执行时间不应无限制）、成本预算（设置单次请求的 token 上限）、幂等性（同一输入重复执行应产生相同结果）、以及优雅降级（当某个角色不可用时，系统应提供有意义的回退而非崩溃）。

另一个生产化要点是配置外部化。角色的系统提示词、模型选择、超时参数、重试策略等都应该通过配置文件管理，而不是硬编码在代码中。这使得你可以不发版就调整系统行为。

### 多智能体系统的调试技巧

多智能体系统的调试比单 Agent 复杂一个数量级，因为 bug 可能出现在任何角色内部、角色之间的交接处、或者 Supervisor 的调度决策中。

有效的调试策略是"分层隔离"。第一步：确认 Supervisor 的路由决策是否符合预期——打印每一步的路由目标和决策依据。第二步：如果路由正确但某个角色的产出有问题，单独测试该角色——用固定的输入调用它，检查输出是否正确。第三步：如果角色单独测试正常但在流程中出问题，检查交接数据是否在传递过程中被修改或截断。

Mock 是多智能体调试的利器。本章中的 `_run_battle_mock` 提供了完整流程的 mock 实现，使得你可以在不消耗 LLM API 的情况下验证编排逻辑的正确性。建议为每个角色都准备 mock 版本，这样可以逐个将角色从 mock 切换为真实实现，精确定位问题所在。

### 常见问题：状态爆炸、级联失败、成本控制

状态爆炸是指共享状态随迭代快速膨胀。在本章的多角色系统中，每轮迭代都会向状态追加 PM 的需求文档、Dev 的实现结果、QA 的测试报告等。如果 Supervisor 决定迭代多次，状态体积可能迅速超出 LLM 的上下文窗口。解决方案包括：状态分区（只将相关角色的历史传给它）、历史压缩（对旧轮次做摘要）、以及外部存储（大块数据存入数据库，状态中只保留引用）。

级联失败是指一个角色的错误沿着链路传播并放大。例如 PM 的需求拆解有误，导致 Dev 实现了错误的功能，QA 基于错误的期望通过了测试，最终 Publisher 发布了有缺陷的产品。防止级联失败的关键是在关键节点设置"断言"——QA 不仅检查 Dev 的实现是否匹配 PM 的需求，还独立验证需求本身是否合理。

成本控制是生产环境中最现实的问题。多角色系统的 LLM 调用次数是单 Agent 的数倍，加上迭代循环可能成倍放大。有效的成本控制手段包括：为不同角色选择不同成本的模型、设置单次请求的总 token 预算、在非关键路径使用缓存（相同输入直接返回历史结果）。

## 新手最容易卡住的点

- 被角色数量吓到——本章有 5 个角色节点加 1 个 Supervisor，看起来复杂，但每个节点的逻辑其实很简单。先理解单个节点做什么，再看它们如何连接。
- 技能注册表与角色的关系——`build_skill_registry` 注册的技能被 Dev 角色调用。理解"谁注册技能"和"谁使用技能"是分开的，有助于理清系统结构。
- 共享状态的读写时机——每个角色从状态中读取什么、写入什么？这是理解整个系统数据流的关键。建议画一张角色-状态交互图。

## 建议动手实验

- 在 QA 节点中加入一个质量评分逻辑，当评分低于阈值时让 Supervisor 路由回 Dev 重做，体会迭代优化的完整循环。
- 为每个角色节点添加执行耗时日志，统计一次完整流程中各角色的时间占比，思考哪些环节可以优化。
- 尝试将 Dev 角色拆分为两个专职角色（如 Frontend Dev 和 Backend Dev），修改图定义以支持并行执行。
- 在不消耗 API 的前提下，先用 mock 模式跑通新增角色的编排逻辑，确认无误后再切换到真实 LLM。
