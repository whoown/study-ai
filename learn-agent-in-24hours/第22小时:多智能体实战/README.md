# 第 22 小时：多智能体实战

## 本章目标

- 把 **第 19–21 小时** 的路线串成一条可运行的「小流水线」：**技能模块化 + 多角色 + 图编排**。
- 用贴近业务语言的例子（**需求澄清 → 开发评估 → 测试清单 → 发布包**）体会如何在代码里落地，而不是停留在概念图。
- 能指出：哪些部分适合继续换成真实 LLM 调用，哪些部分应保持确定性函数以便测试。

## 核心概念

- **组合优于堆叠**：Skills 负责「小而稳的工具能力」；各 `*_node` 负责「角色视角的推理与产出」；LangGraph 负责「谁在什么时候上场」。
- **演进回顾**
  - 第 19 小时：`Skill` + 注册表 + 单调度器。
  - 第 20 小时：多个角色函数与共享状态。
  - 第 21 小时：`supervisor` 条件边驱动的编排。
  - **第 22 小时：把技能嵌进节点里，跑通端到端案例。**

## 案例设计

用户输入一句「想法/需求」，系统依次经过：

1. **PM 节点**：输出结构化 PRD 摘要（模板或可选模型）。
2. **Dev 节点**：调用技能 `estimate_effort` 给出粗估人天区间。
3. **QA 节点**：调用技能 `build_test_checklist` 生成测试要点。
4. **Publisher 节点**：汇总为对外可读的 `delivery` 文本。

若缺少 `langgraph` 或 API Key，程序分别降级为 **模拟图执行** 与 **模板文本**，终端会给出说明。

## 代码讲解

文件：`src/main.py`。

- **`Skill` / `run_skill`**：与第 19 小时同构的轻量技能层，本章在 Dev/QA 节点内调用。
- **`BattleState`**：端到端共享状态，贯穿所有节点。
- **`pm_node` / `dev_node` / `qa_node` / `publisher_node`**：四个智能体节点，体现分工。
- **`supervisor_node` + `_route_supervisor`**：与第 21 小时相同的编排手法，把四个节点串进一张图。
- **`run_battle`**：统一入口，`compile()` 后 `invoke`；失败时走 `_run_battle_mock`。

## 运行方式

```bash
cd "第22小时:多智能体实战"
python src/main.py
```
