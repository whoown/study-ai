# 第 16 小时：规划执行

## 本章定位

先拟计划，再分步执行。本章介绍 Plan-and-Execute 范式——一种将复杂任务分解为可管理步骤的 Agent 架构。与 ReAct 的"边做边想"不同，Plan-and-Execute 要求 Agent 先制定完整计划，再按计划逐步执行，适合处理需要多步骤协调的复杂任务。

## 核心概念

### Plan-and-Execute 范式的核心思想

Plan-and-Execute 是一种将 Agent 的工作分为两个明确阶段的架构：先由一个"规划器"（Planner）将用户目标分解为有序的步骤序列，再由一个"执行器"（Executor）逐步执行这些步骤。

这种范式背后的直觉来自人类处理复杂任务的方式——面对"组织一次团队旅行"这样的任务，我们不会走一步看一步，而是先列出清单（订票、订酒店、规划行程、通知成员……），再逐项完成。对 Agent 而言，同样如此：复杂任务如果不经过规划就直接执行，容易陷入局部最优、遗漏关键步骤、或在步骤间产生冲突。

Plan-and-Execute 的核心优势在于 **全局视野**。规划阶段可以纵览整个任务，考虑步骤之间的依赖关系和优先级，避免执行阶段的短视行为。此外，计划本身也是一种可解释的中间产物——用户可以审查计划、提出修改，实现更好的人机协作。

### 计划生成（Planning）的方式

计划生成主要有两种范式。

**一次性计划（One-shot Planning）** 在任务开始前一次性生成完整的步骤序列。这种方式适合任务结构明确、步骤间依赖关系可预判的场景。优点是全局最优性更好、执行前可以预审；缺点是对不确定性的处理能力弱——如果某个步骤的结果影响后续步骤的方向，一次性计划无法适应。

**渐进式计划（Incremental Planning）** 只规划接下来的几步，执行后根据结果再规划下一阶段。这种方式更灵活，能根据执行反馈动态调整方向。缺点是缺少全局视野，可能导致步骤间的协调不足。

实践中，一种常见的折中方案是"粗规划 + 细执行"——先生成一个粗粒度的全局计划，在执行每个粗粒度步骤时再动态细化为具体操作。

计划的质量高度依赖于 prompt 设计。好的规划 prompt 应该明确要求：每个步骤必须可执行（而非模糊的描述）、步骤之间必须标注依赖关系、估算每个步骤的产出物。

### 步骤间的依赖关系（Dependency）管理

复杂任务中的步骤往往不是线性排列的，而是存在依赖关系——某些步骤必须等待其他步骤完成后才能开始。

**显式依赖** 是指在计划中直接标注每个步骤依赖哪些前置步骤。例如"步骤 3 依赖步骤 1 和步骤 2 的输出"。这使得执行器可以准确判断哪些步骤可以并行执行、哪些必须串行等待。

**隐式依赖** 是指步骤间的依赖关系没有显式标注，需要从步骤描述中推断。这种情况更常见但也更容易出错——执行器可能遗漏依赖关系，导致在前置条件未满足时就执行了某个步骤。

依赖管理的数据结构通常是一个有向无环图（DAG），而非简单的线性序列。在 DAG 中，没有依赖关系的步骤可以并行执行，有依赖的步骤则必须等待所有前置步骤完成。检测依赖图中是否存在环（死锁）是执行前的必要校验步骤。

### 计划执行（Execution）中的错误处理与重试

执行阶段面临的核心挑战是：真实环境充满不确定性，任何步骤都可能失败。健壮的执行器需要具备完善的错误处理机制。

**重试策略** 是最基本的容错手段。对于暂时性故障（如网络超时、API 限流），简单重试通常能解决问题。重试应配合退避策略（如指数退避），避免在故障期间频繁请求加剧问题。

**跳过与降级** 适用于非关键步骤的失败。如果某个步骤是可选的增强步骤，失败后可以跳过而不影响整体流程的完成。

**错误传播与终止** 是对关键步骤失败的处理。如果一个步骤是后续多个步骤的前提，其失败意味着整个分支都无法继续。执行器需要能识别这种情况并及时终止，而非在无效的上下文上继续执行。

每个步骤执行后应记录结果（成功/失败/输出），这些记录既用于后续步骤的输入，也为错误诊断和重规划提供信息。

### 动态重规划（Re-planning）

静态计划在执行过程中可能因为环境变化或步骤失败而不再适用。动态重规划允许 Agent 在执行中途修正计划。

触发重规划的常见条件包括：关键步骤失败且重试无效、步骤执行结果与预期严重不符、发现了规划时未考虑到的新信息、用户在执行过程中修改了目标。

重规划时需要保留已完成步骤的成果，只对未完成部分重新规划。这要求计划的数据结构能清晰记录每个步骤的状态（待执行/执行中/已完成/已失败），以及已完成步骤产生的中间结果。

重规划的频率需要控制——过于频繁的重规划意味着原始计划质量不高或任务本身不适合 Plan-and-Execute 范式。如果发现大量任务需要频繁重规划，可能应该回退到 ReAct 等更灵活的范式。

### Plan-and-Execute vs ReAct 的对比

ReAct（Reasoning + Acting）和 Plan-and-Execute 是两种主流的 Agent 架构，适用于不同类型的任务。

**ReAct** 采用"思考-行动-观察"的循环，每一步都基于当前状态决定下一步行动。优势是灵活性高、能快速响应环境变化；劣势是缺乏全局视野、可能在复杂任务中"迷路"。

**Plan-and-Execute** 先规划后执行，有全局视野和结构化的执行流程。优势是对复杂多步任务的处理更可靠、中间状态更可控；劣势是规划阶段有额外开销、对动态环境的适应性不如 ReAct。

选择哪种范式取决于任务特征：步骤间有明确依赖关系、任务结构可预判的用 Plan-and-Execute；步骤间高度动态、需要频繁根据反馈调整的用 ReAct；两者也可以混合——在整体框架上用 Plan-and-Execute，在每个步骤的内部执行中用 ReAct。

### 规划执行的常见问题

**计划过粗** 指步骤描述过于笼统，执行器无法直接转化为具体操作。例如"收集相关信息"作为一个步骤就过于模糊。解决方式是在规划 prompt 中要求每个步骤必须包含明确的输入、操作和输出。

**计划过细** 指将任务分解为过多的细碎步骤，增加了执行开销和出错概率。通常是规划器对任务理解过度导致。可以通过限制计划步骤数量或引入层级规划来缓解。

**依赖死锁** 出现在步骤间的依赖关系形成环路时（A 依赖 B，B 依赖 A）。虽然正确设计的系统不应产生环路，但在 LLM 生成计划时这种错误偶有发生。执行前应进行拓扑排序验证。

**执行偏移** 指执行过程中逐步偏离原始计划目标。每个步骤的小偏差累积后可能导致最终结果与预期目标严重不符。定期对照原始目标审查执行进度是必要的纠偏手段。

## 新手最容易卡住的点

- 将 `build_plan` 生成的计划视为不可变——实际上计划应该能在执行过程中被修改和调整。
- 忽略 `can_run_step` 中的依赖检查逻辑——没有依赖管理的计划执行本质上只是顺序执行脚本，不具备 Agent 的容错能力。
- 把 `fake_tool_call` 的模拟执行当成真实场景——真实环境中每一步都可能失败，需要设计错误处理路径。
- 混淆"计划"和"执行"两个阶段的职责——规划器只负责"做什么"，执行器负责"怎么做"。

## 建议动手实验

- 修改 `build_plan` 的输入目标，观察生成的计划步骤和依赖关系如何变化。
- 在 `fake_tool_call` 中随机注入失败，观察 `run_plan_and_execute` 的错误处理行为。
- 手动修改生成计划中的依赖关系，制造一个循环依赖，观察 `can_run_step` 是否能检测到死锁。
- 对比同一个任务分别用 3 步和 10 步计划来完成，思考合理的计划粒度。
