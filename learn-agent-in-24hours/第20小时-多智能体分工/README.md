# 第 20 小时：多智能体分工

## 本章定位

把任务拆给不同角色协作完成。单个 Agent 即使工具再多、上下文再长，也会在复杂任务面前遇到瓶颈——提示词越来越长、注意力被稀释、错误难以定位。本章通过引入多个专职 Agent（Planner、Researcher、Writer），将一个复杂任务拆解为多个子任务，每个 Agent 只专注自己的职责，最终通过流水线串联完成协作。

## 核心概念

### 多智能体系统的基本概念

多智能体系统（Multi-Agent System, MAS）是指由多个自主 Agent 组成的系统，这些 Agent 各自拥有独立的目标、知识和能力，通过协作来完成单个 Agent 难以独立完成的任务。与单 Agent 多工具模式不同，多智能体系统中的每个 Agent 有自己独立的系统提示词、独立的推理过程、甚至可以使用不同的底层模型。

为什么要引入多智能体？根本原因是"认知负荷"问题。当你把规划、检索、写作、校对、格式化等所有职责都塞进一个 Agent 的提示词里，模型的注意力机制会被过度分散，指令遵循的准确率会显著下降。将不同职责拆分到不同 Agent，每个 Agent 的提示词更短更聚焦，推理质量自然更高。

此外，多智能体架构还带来了工程上的好处：各角色可以独立迭代和测试，出现问题时可以精确定位到具体角色，不同角色甚至可以使用不同成本/能力的模型（如 Planner 用强模型，Writer 用快模型）。

### 角色（Role）设计原则

角色设计是多智能体系统成败的关键。一个好的角色设计应遵循以下原则。

单一职责原则：每个角色只做一件事，且做好这一件事。本章中 Planner 只负责拆解任务、Researcher 只负责搜集信息、Writer 只负责撰写输出。如果一个角色同时承担了规划和执行，它的表现通常不如两个专职角色的组合。

能力互补原则：角色之间的能力应该形成互补而非重叠。Planner 需要全局视野和逻辑推理能力，Researcher 需要信息检索和事实核查能力，Writer 需要语言组织和格式化能力。重叠的能力意味着冗余的计算和潜在的冲突。

清晰的输入输出定义：每个角色应该有明确的输入格式和输出格式。Planner 接收原始任务描述，输出子任务列表；Researcher 接收子任务列表，输出事实信息；Writer 接收事实信息，输出最终文本。这种清晰的接口约定使得角色可以被独立测试和替换。

### 流水线（Pipeline）编排模式

流水线是最简单也最直观的多智能体编排模式：Agent A 的输出作为 Agent B 的输入，Agent B 的输出再作为 Agent C 的输入，依此类推。本章的 `run_pipeline` 就是这种模式的实现。

流水线模式的优势在于简单和可预测。执行顺序是确定的，数据流向是单向的，任何一个环节出问题都可以立即定位。对于"先规划、再调研、最后写作"这类天然具有顺序依赖的任务，流水线是最佳选择。

但流水线也有明显的局限。它假设任务可以被线性分解，不允许后续环节反馈给前序环节。如果 Writer 发现 Researcher 提供的信息不足，它没有办法要求 Researcher 补充——整个流水线只能向前推进。这个局限将在后续章节（编排和循环）中被解决。

### Hand-off（交接）机制

Hand-off 是指一个 Agent 将工作成果和必要的上下文传递给下一个 Agent 的过程。看似简单，但交接机制的设计直接决定了多智能体系统的运行质量。

交接的核心问题是"传什么"和"怎么传"。最简单的方式是传递纯文本——上一个 Agent 的输出文本直接作为下一个 Agent 的输入。但在实际场景中，你往往需要传递结构化数据：任务列表、事实条目、元数据标签等。结构化的交接数据使得下游 Agent 可以精确地处理每一项内容，而不需要从自由文本中重新解析。

另一个关键设计点是上下文传递的范围。是只传递上一步的输出，还是把整条链路的历史都带上？全量传递保证了信息完整性，但会导致上下文膨胀；最小化传递节省了 token，但可能丢失关键信息。实践中常用的策略是"摘要传递"——每个 Agent 输出一份简明摘要，附带完整数据的引用链接。

### 协调（Coordination）策略

即使在流水线这种最简单的编排模式下，协调策略也是不可回避的话题。协调需要回答几个问题：谁来决定执行顺序？如何处理某个 Agent 的失败？如何控制整体执行的超时？

在本章中，协调逻辑内嵌在 `run_pipeline` 函数中——它硬编码了 Planner → Researcher → Writer 的顺序。这是最简单的静态协调。更灵活的协调方式包括：基于条件的分支（如果 Planner 判断任务简单就跳过 Researcher）、基于事件的触发（某个 Agent 发布事件后才启动下一个）、以及基于优先级的调度（多个 Agent 竞争执行权）。

错误处理是协调中最容易被忽视的环节。当 Researcher 调用外部 API 失败时，是重试、跳过还是终止整个流水线？合理的做法是为每个角色定义降级策略，并在协调层实现统一的错误处理逻辑。

### 多智能体 vs 单 Agent 多工具的取舍

并非所有场景都需要多智能体。如果任务简单、步骤少、不需要深度推理，一个 Agent 配几个 Tool 就足够了。引入多智能体会增加系统复杂度、通信开销和 LLM 调用成本。

经验上的判断标准是：当你发现单 Agent 的提示词超过 2000 字、或者需要在提示词中写"先做 X，然后做 Y，接着做 Z"这种长流程描述时，就该考虑拆分为多智能体。另一个信号是当你发现 Agent 频繁"遗忘"早期指令或在多步任务中表现不稳定时——这通常意味着单 Agent 的认知负荷已经过高。

### 角色数量与粒度的设计经验

角色不是越多越好。每增加一个角色就增加一次 LLM 调用和一次交接，且角色间的协调复杂度随角色数量呈超线性增长。实践中的经验法则是：3-5 个角色适合大多数场景，超过 7 个角色的系统通常需要引入分层管理（即 Supervisor 模式，这是下一章的主题）。

角色粒度的把握同样重要。粒度太粗（如一个"全能助手"角色），等于没有拆分；粒度太细（如"标点符号检查员"角色），则增加了不必要的通信开销。好的粒度应该让每个角色对应一个"可独立评估质量"的产出物——Planner 产出的计划可以被评估、Researcher 收集的信息可以被核查、Writer 的文章可以被审阅。

## 新手最容易卡住的点

- 混淆"多工具"和"多智能体"——如果三个函数共享同一份系统提示词，那只是一个 Agent 在用三个工具，而不是三个 Agent 在协作。检查每个角色是否有独立的 system prompt 是判断标准。
- 交接数据格式不一致——Planner 输出的是自由文本，但 Researcher 期待的是结构化列表，这种不匹配会导致下游 Agent 理解偏差。确保每一步交接的输入输出格式有明确约定。
- 忽视了流水线的单向性局限——如果你的任务需要迭代优化（比如 Writer 写完后要回到 Researcher 补充信息），纯流水线是做不到的，需要后续章节的编排能力。

## 建议动手实验

- 在 Planner 和 Writer 之间插入一个新的 Reviewer 角色，让它审核 Researcher 的输出质量，体会角色增加对流程的影响。
- 故意让 Researcher 返回空结果，观察 Writer 在信息不足时的降级行为。
- 对比同一个任务在单 Agent（把三个角色的提示词合并）和多 Agent 模式下的输出质量差异。
- 修改交接数据格式（如从纯文本改为 JSON 结构），观察对下游角色理解准确度的影响。
