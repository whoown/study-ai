# 第 2 小时：接入真实 LLM

## 本章定位

将上一章的硬编码规则引擎替换为真实的大语言模型，让 Agent 获得通用的自然语言理解与推理能力。本章通过 `chat_completion` 实现真实 API 调用，同时保留 `mock_chat_completion` 离线分支，确保在无网络环境下也能完整走通流程。这一步是从"玩具 Agent"迈向"可用 Agent"的关键跨越。

## 核心概念

### LLM 作为 Agent 大脑的角色

在上一章中，Agent 的决策层是一个 `fake_brain` 函数——通过 if-else 规则返回预设的行动指令。这种方式虽然可靠且可预测，但能力极其有限：它只能处理开发者预先想到的场景。而大语言模型（LLM）的引入，从根本上改变了决策层的能力边界。

LLM 作为 Agent 的大脑，承担的核心职责是：**理解当前上下文，生成合理的下一步行动计划**。与规则引擎不同，LLM 具备泛化能力——它可以处理训练数据中从未出现过的输入组合，通过语义理解来推断应该采取什么行动。这种能力使得 Agent 不再局限于预定义场景，而是可以应对开放域问题。

但 LLM 作为决策层也引入了新的挑战：**不确定性**。规则引擎对同一输入永远返回同一输出，而 LLM 的输出具有随机性（由 temperature 等参数控制）。同一个问题问两次，可能得到格式不同、甚至内容不同的回答。这意味着 Agent 的循环逻辑必须具备更强的鲁棒性——能够处理各种格式的模型输出、在模型"答非所问"时有兜底策略。

从架构角度看，LLM 是 Agent 中最昂贵的组件。每次调用都涉及网络请求、计算资源和 API 费用。因此，减少不必要的 LLM 调用、优化 Prompt 以提高一次成功率、在可能的情况下缓存结果，都是实际工程中需要关注的优化点。

### System Prompt 的设计原则

System Prompt 是 Agent 开发中最重要的工程产物之一。它定义了 LLM 的角色、行为边界和输出格式，直接决定了 Agent 的表现质量。一个设计糟糕的 System Prompt 会让 Agent 频繁产生不可解析的输出，而一个精心设计的 Prompt 可以让 Agent 在大多数情况下稳定工作。

System Prompt 的设计遵循几个关键原则。第一是**角色定义清晰**：明确告诉模型它是什么角色、要完成什么任务。模糊的角色定义会导致模型"漫游"——在多个可能的解读之间摇摆不定。第二是**输出格式明确**：Agent 的后续流程需要解析模型输出，因此必须在 Prompt 中严格约定输出格式。如果你需要模型返回 JSON，就在 Prompt 中给出 JSON 示例；如果你需要特定的标记（如 `ACTION:` 前缀），就明确要求并给出示范。

第三个原则是**约束优于自由**。LLM 天然倾向于生成冗长、发散的内容。在 Agent 场景中，你需要的是精确、简洁、格式化的输出。因此，Prompt 中应该明确列出模型"不应该做什么"，比如"不要解释你的推理过程""不要添加额外的寒暄内容"等否定约束往往比正面指令更有效。

第四个原则是**示例驱动**（Few-shot）。与其用长篇文字描述你想要的输出格式，不如直接给出 2-3 个输入输出示例。LLM 在模式匹配方面能力极强，几个高质量的示例通常比详尽的规则描述效果更好。

### API 调用模式（Chat Completion）

Chat Completion 是当前主流 LLM API 的标准交互模式。它的核心数据结构是一个**消息列表**（messages），每条消息包含角色（role）和内容（content）两个字段。角色通常有三种：`system`（系统指令）、`user`（用户输入）、`assistant`（模型回复）。

理解 Chat Completion 的关键在于认识到它是**无状态的**。每次 API 调用都是独立的——模型不会"记住"上一次调用的内容。如果你需要模型看到对话历史，必须在每次请求中把完整的历史消息都带上。这意味着 Agent 循环中的上下文管理完全由客户端负责：每一轮循环结束后，你需要把模型的回复和工具的 Observation 追加到消息列表中，下一轮调用时带上完整列表。

这种无状态设计带来一个直接的工程问题：**上下文窗口限制**。每个模型都有最大 token 数限制，当对话历史过长时，你必须做截断或摘要。这个问题在本章还不明显（因为对话轮次少），但在后续构建复杂 Agent 时会成为核心挑战之一。

API 调用还涉及几个重要参数：`temperature` 控制输出的随机性（Agent 场景通常设为较低值以保证稳定性）、`max_tokens` 控制输出长度、`stop` 设置停止标记。合理配置这些参数对 Agent 的稳定运行至关重要。

### Fallback 降级策略的工程意义

Fallback（降级）是指当主路径不可用时，系统自动切换到备用路径继续工作的机制。在 Agent 开发中，LLM API 的不可靠性使得 Fallback 不是"锦上添花"，而是"必须具备"的工程能力。

LLM API 可能因为多种原因失败：网络超时、API 限流（rate limiting）、服务端故障、返回格式异常等。如果 Agent 在 API 调用失败时直接崩溃，那它在生产环境中就是不可用的。本章的 `mock_chat_completion` 就是最基本的 Fallback 实现：当真实 API 不可用时，使用预设的模拟响应来保证流程可以继续。

实际生产中的 Fallback 策略更加多样：**重试机制**（带指数退避的自动重试）、**模型降级**（主模型不可用时切换到备用模型）、**缓存命中**（对相似请求返回缓存结果）、**静态规则兜底**（极端情况下回退到规则引擎）。这些策略通常是分层组合使用的，形成一个从"最优"到"可用"的优雅退化路径。

Fallback 设计还涉及一个重要的权衡：**可用性 vs 准确性**。降级后的响应质量通常低于主路径，但至少保证了系统不会完全停摆。在设计 Agent 时，你需要明确在什么场景下"给出一个可能不完美的回答"优于"完全不回答"，并据此设计你的降级层次。

## 新手最容易卡住的点

- **API Key 配置问题**：第一次接入真实 LLM 时，最常见的失败原因是 API Key 未设置或设置错误。确保环境变量或配置文件中的 Key 正确无误。
- **混淆有状态和无状态**：以为调用了一次 API 后模型就"记住"了上下文。实际上每次调用都是独立的，必须手动维护消息历史。
- **模型输出格式不稳定**：模型可能不按预期格式输出，导致 `parse_plan` 解析失败。这不是 bug，而是 LLM 的本质特征，需要通过更好的 Prompt 和更鲁棒的解析逻辑来应对。
- **忽略离线分支的价值**：`mock_chat_completion` 不只是"没有 API Key 时的替代品"，它也是调试循环逻辑时的利器——用确定性输出来排查循环流程中的问题。

## 建议动手实验

- 修改 System Prompt 的内容，观察模型输出风格和格式的变化，体会 Prompt 对 Agent 行为的控制力。
- 故意传入一个错误的 API Key，观察系统如何降级到 mock 分支，理解 Fallback 的触发机制。
- 在 `agent_loop` 中打印每一轮的完整消息列表（messages），观察上下文是如何逐轮累积的。
- 调整 `temperature` 参数（比如从 0 改为 1），多次运行同一输入，观察输出的变化程度。
