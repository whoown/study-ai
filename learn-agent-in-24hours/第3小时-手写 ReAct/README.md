# 第 3 小时：手写 ReAct

## 本章定位

显式写出 Thought-Action-Observation 三段式循环，让 Agent 在每一步行动前先进行推理思考。本章通过 `react_loop` 核心循环和 `parse_react_block` 解析逻辑，手动实现了 ReAct 框架的完整流程。这是 Agent 从"能用"到"可控可解释"的重要一步——思考过程被外显出来，使 Agent 的行为不再是黑盒。

## 核心概念

### ReAct 框架的起源与论文思想

ReAct（Reasoning + Acting）框架来源于 2022 年普林斯顿大学和 Google 联合发表的论文《ReAct: Synergizing Reasoning and Acting in Language Models》。这篇论文的核心洞察是：**让 LLM 在行动之前显式地进行推理，可以显著提升复杂任务的完成质量**。

在 ReAct 之前，LLM 的使用主要分为两个独立的方向：一个是"推理"方向（如 Chain-of-Thought），让模型一步步思考来解决数学题、逻辑题；另一个是"行动"方向，让模型直接输出需要执行的操作。ReAct 的创新在于将这两个方向统一到一个交替进行的循环中：Thought → Action → Observation → Thought → Action → ...

这种统一带来了两个关键好处。第一，**推理指导行动**：模型在行动前先分析当前情况、规划策略，这减少了盲目试错。第二，**行动结果反馈推理**：工具返回的 Observation 为下一轮推理提供了新信息，使模型能够根据实际结果动态调整策略，而不是固守最初的计划。

从工程角度看，ReAct 框架还有一个被低估的优势：**可调试性**。由于思考过程被显式记录为文本，开发者可以在 Agent 出错时回溯每一步的推理，找到是"想错了"还是"做错了"，这在纯行动模式中是做不到的。

### Thought（思考）步骤的作用

Thought 步骤是 ReAct 框架中最具特色的部分。它要求模型在采取任何行动之前，先用自然语言表达当前的分析和推理。这看起来是一个简单的要求，但它对 Agent 行为产生了深远的影响。

Thought 的首要作用是**激活模型的推理能力**。LLM 的推理能力并非自动触发的——直接问模型"下一步做什么"，模型可能会跳过分析直接给出一个表面合理的答案。而要求模型先"写下想法"，实际上是在强制它进行更深入的信息处理。这与 Chain-of-Thought Prompting 的原理一致：让模型"说出思考过程"本身就能提升回答质量。

Thought 的第二个作用是**提供上下文锚点**。在多轮循环中，每一步的 Thought 记录了模型对当前状态的理解。当这些 Thought 被追加到消息历史中，后续的推理可以参考之前的思考，避免重复分析或遗忘关键信息。

Thought 的第三个作用是**约束行动空间**。当模型先"想清楚"再行动时，它更倾向于选择与当前分析一致的行动，而不是随机尝试。这使得 Agent 的行为序列更连贯、更高效。

在实际开发中，Thought 步骤也有其成本：它增加了每轮循环的 token 消耗和延迟。对于简单任务，Thought 可能是"多余的开销"；对于复杂任务，它则是"必要的投资"。如何在不同场景中权衡，是 Agent 优化的一个重要方向。

### Action（行动）与工具调用的关系

Action 是 ReAct 循环中模型"做事"的步骤。在本章的实现中，Action 以特定格式的文本呈现（如 `Action: tool_name(arg1, arg2)`），由 `parse_react_block` 解析后交给对应的工具函数执行。

Action 与第 1 章的工具调用在本质上是相同的——都是 Agent 通过工具与外部环境交互。但在 ReAct 框架中，Action 获得了更丰富的上下文：它不再是孤立的一次调用，而是"在 Thought 的推理指导下"选择的行动。这种上下文关联让 Action 的选择更加合理。

Action 的解析是 ReAct 手动实现中最脆弱的环节。模型输出的文本可能不完全遵循预定格式：多余的空格、缺少括号、参数顺序错误、甚至幻觉出不存在的工具名——这些都是实际开发中常见的问题。`parse_react_block` 需要具备足够的容错能力，能在模型输出"基本正确但不完美"时仍然提取出有效信息。

一个常见的工程决策是：当 Action 解析失败时该怎么办？选项包括：将解析错误作为 Observation 返回给模型，让它修正格式后重试；直接跳过这一轮，进入下一轮推理；或者达到连续解析失败上限后终止循环。不同的策略适用于不同的场景，但核心原则是**不要让一次解析失败导致整个 Agent 崩溃**。

### Observation（观察）如何驱动下一轮推理

在 ReAct 框架中，Observation 的角色比第 1 章更加核心。它不仅是工具的返回结果，更是驱动下一轮 Thought 的关键输入。Observation 回答的问题是："我刚才那个行动产生了什么效果？"模型基于这个答案来决定是否需要继续行动、改变策略、还是任务已经完成。

Observation 的驱动机制可以这样理解：Thought 产生期望（"如果我调用搜索工具，应该能找到 X 信息"），Action 执行操作，Observation 返回实际结果。如果结果符合期望，Agent 沿着当前策略继续；如果结果不符合期望（比如搜索没有找到有用信息），Agent 需要在下一轮 Thought 中调整策略。这种"预期—执行—验证—调整"的循环是 ReAct 框架能够处理复杂任务的根本原因。

Observation 的格式设计也很重要。过于简略的 Observation（如仅返回 `true/false`）会让模型缺少做决策的信息；过于冗长的 Observation（如返回整个数据库查询结果）会占用大量上下文窗口并可能干扰推理。理想的 Observation 应该**简洁但信息完整**——包含模型做下一步决策所需的关键信息。

### ReAct vs 纯 Chain-of-Thought 的区别

Chain-of-Thought（CoT）和 ReAct 都让模型进行显式推理，但它们有一个本质区别：CoT 是"在脑子里想完所有步骤再给出答案"，而 ReAct 是"想一步、做一步、看反馈、再想下一步"。

CoT 的局限在于它只能利用模型的内部知识。如果答案需要实时数据、精确计算或外部信息，纯 CoT 就力不从心了。模型可能会"编造"一个看起来合理但实际错误的答案，因为它无法验证自己的推理。

ReAct 通过引入 Action 和 Observation，让模型能够在推理过程中获取外部信息。模型不需要"猜"一个数字的乘积，它可以调用计算工具获得准确结果；不需要"凭记忆"回答实时问题，它可以调用搜索工具获取最新数据。这种"推理 + 外部验证"的组合，大幅提升了 Agent 在真实世界任务中的可靠性。

当然，ReAct 也有其代价：多轮循环意味着更多的 API 调用、更长的执行时间、更高的成本。对于纯推理任务（如数学证明），CoT 可能更高效；对于需要信息检索和外部操作的任务，ReAct 是更好的选择。在实践中，很多高质量的 Agent 系统会根据任务类型动态选择推理模式。

## 新手最容易卡住的点

- **把 Thought 当作"可有可无的注释"**：Thought 不是装饰，它是推理能力的触发器。删掉 Thought 步骤，Agent 的表现通常会明显下降。
- **解析逻辑太脆弱**：`parse_react_block` 需要能容忍模型输出的微小格式偏差。如果你的解析只能处理"完美格式"，在实际使用中会频繁报错。
- **不理解 Observation 如何反馈到 Thought**：关键在于 Observation 被追加到消息历史中，下一轮 Thought 是在包含了 Observation 的上下文中生成的。
- **循环没有设置上限**：如果模型持续输出无法解析的 Action，循环会无限进行。务必设置 max_steps 作为安全阀。

## 建议动手实验

- 在 System Prompt 中删掉对 Thought 步骤的要求，只保留 Action 和 Observation，运行后对比 Agent 行为的变化。
- 故意让 `parse_react_block` 更严格（比如不容忍任何空格偏差），观察 Agent 在实际运行中的失败频率。
- 在每轮循环中打印完整的 Thought → Action → Observation 序列，追踪模型的推理链路是否连贯。
- 设计一个需要 3 步以上工具调用才能完成的任务，观察 ReAct 循环如何逐步推进。
