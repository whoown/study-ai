# 第 7 小时：短期记忆

## 本章定位

让 Agent 记住当前会话里的上下文。前面几章的 Agent 每次调用都是"失忆"的——它看不到之前说过什么、做过什么。本章引入短期记忆机制，使 Agent 能在一次会话内维持连贯的多轮对话，为后续的上下文管理和长期记忆打下基础。

## 核心概念

### Memory 系统的分类：短期记忆 vs 长期记忆

Agent 的记忆系统可以类比人类的记忆模型，分为短期记忆（Short-term Memory）和长期记忆（Long-term Memory）两大类。

短期记忆是指在单次会话期间维持的上下文信息，它的生命周期等同于一次对话 session。当会话结束，短期记忆即被丢弃。最常见的实现方式是将完整的消息历史（messages list）保留在内存中，每次调用模型时将其完整传入。短期记忆的优势在于实现简单、信息无损，但受限于模型的上下文窗口大小。

长期记忆则跨越会话边界，通常需要外部存储支撑——例如数据库、向量库或文件系统。它适合存放用户偏好、历史交互摘要、学到的技能等需要持久化的信息。长期记忆的核心挑战在于检索效率和信息的时效性管理。

理解这两者的区分至关重要：短期记忆解决的是"当前对话的连贯性"问题，长期记忆解决的是"跨会话的知识积累"问题。本章专注于短期记忆的实现。

### Messages 列表作为短期记忆的实现方式

在基于大语言模型的 Agent 中，短期记忆最自然的载体就是 messages 列表。每一轮对话都会产生新的消息条目，这些条目按时间顺序追加到列表中，形成一条完整的对话链。

这种实现方式有几个关键特性：第一，它是有序的，消息的先后顺序本身携带了因果和时序信息；第二，它是累加的，每一轮对话都在已有历史上追加新消息；第三，它是透明的，模型能直接"看到"历史消息的原文，不存在信息压缩导致的失真。

在实际实现中，`rebuild_messages` 函数在每次模型调用前重新构建完整的消息列表。这种"重建"而非"追加"的设计模式具有重要意义——它使得开发者能在每次调用前对消息列表进行灵活的干预，比如插入系统提示、过滤无关消息、调整消息顺序等。

### 角色（Roles）在消息中的作用

消息列表中的每条消息都携带一个角色标识（role），这是 LLM 理解对话结构的关键机制。主要有四种角色：

**system** 角色用于设定 Agent 的行为边界和人格特征。系统消息通常位于消息列表的最前面，它不参与对话的信息交换，而是作为一种"元指令"持续影响模型的所有后续输出。好的系统提示是 Agent 表现稳定的基石。

**user** 角色代表人类用户的输入，是驱动对话前进的主要力量。**assistant** 角色是模型自身的输出，包括文本回复和工具调用请求。**tool** 角色用于返回工具执行的结果，它使模型能感知外部操作的反馈。

这四种角色构成了一个完整的信息流转闭环：system 定调 → user 提问 → assistant 思考/调用工具 → tool 返回结果 → assistant 继续推理。理解这个闭环是构建多轮 Agent 的前提。

### 记忆回放（Replay）机制与多轮对话

LLM 本身是无状态的——每次 API 调用都是独立的，模型并不"记得"上一次调用的内容。多轮对话的效果完全依赖于客户端的记忆回放机制：将之前所有的对话历史重新传入模型，让模型在"重新阅读"历史的过程中恢复上下文。

这意味着每次模型调用的计算成本会随着对话轮次的增加而线性增长。第一轮对话可能只传入几百个 token，到第二十轮可能需要传入上万个 token。这种回放机制虽然简单可靠，但它直接引出了上下文窗口管理的问题——这正是下一章要解决的课题。

在实际的多轮会话实现中，`run_session_with_memory` 函数承担了会话循环的驱动职责：接收用户输入、追加到消息列表、调用模型、处理工具调用、再次调用模型，如此往复直到对话结束。

### 记忆系统开发中的常见问题

**上下文膨胀**是最典型的问题。随着对话的进行，消息列表不断增长，最终会超过模型的上下文窗口限制，导致 API 报错或被截断。在工具密集型场景中，这个问题尤为突出——一次工具调用往往会产生较长的结果文本，几轮下来 token 消耗就可能激增。

**信息丢失**是另一个常见问题。当不得不截断历史消息时，如何保证关键信息不被丢弃？简单的"删除最早消息"策略可能会导致 Agent 忘记用户在对话开头提出的核心需求。

**角色一致性**也容易出问题。如果系统提示在记忆重建过程中被意外修改或遗漏，Agent 的行为可能会发生不可预期的漂移。`build_base_system` 函数通过每次重建时固定注入系统提示来保障这一点。

### 记忆系统的优化方向

当短期记忆面临容量瓶颈时，有几种常见的优化策略：

**压缩**：对历史消息进行摘要，用一段简短的总结替代冗长的原始对话。这种方式能显著减少 token 消耗，但会丢失细节信息。

**选择性遗忘**：根据消息的重要性进行优先级排序，优先丢弃低价值的消息（如闲聊、重复确认等），保留高价值的消息（如关键决策、用户需求陈述等）。

**分层存储**：将近期消息保持原文，将较早的消息做摘要压缩，形成"近期详细、远期概括"的分层结构。这种策略在信息保留和 token 消耗之间取得了较好的平衡，是目前工业实践中被广泛采用的方案。

这些优化方向将在下一章"上下文管理"中得到具体展开。

## 新手最容易卡住的点

- 混淆"模型有记忆"和"客户端传入了历史"。LLM 本身是无状态的，所有"记忆"效果都来自消息列表的重新传入。如果发现 Agent "忘了"之前的对话，首先检查是否正确维护了 messages 列表。
- 忽略 system 消息的持久性。system 消息应该在每次调用时都存在于消息列表的开头，而不是只在第一次调用时传入。
- 工具调用消息的配对问题。assistant 的 tool_call 消息和 tool 的结果消息必须成对出现且顺序正确，缺失或错位都会导致模型困惑甚至 API 报错。
- 消息列表的打印调试。当 Agent 表现异常时，最有效的调试手段是直接打印出传给模型的完整 messages 列表，检查其中是否有遗漏、重复或顺序错误。

## 建议动手实验

- 在 `run_session_with_memory` 的循环中打印每轮传给模型的 messages 列表长度，观察它如何随着对话轮次线性增长。
- 尝试在对话第三轮时引用第一轮的内容（比如"我刚才问的第一个问题是什么"），验证短期记忆是否生效。
- 故意删除 `rebuild_messages` 中对 system 消息的注入，观察 Agent 行为会发生什么变化。
- 让对话进行足够多轮，直到触发 token 超限错误，体会上下文膨胀的实际影响。
