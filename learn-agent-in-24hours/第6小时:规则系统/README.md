# 第 6 小时：规则系统

## 本章目标

- 理解：**LLM 不是唯一决策源**；工程上常用「规则 + 模型」组合。
- 实现一个最小 **RuleEngine**：在调用模型前/后对文本做拦截、改写或注入约束。
- 把规则显式化，便于测试、审计与迭代（比把一切都写进 prompt 更可控）。

## 核心概念

- **前置规则（Pre-rules）**：用户输入进入模型前执行，例如敏感词拦截、格式归一化。
- **后置规则（Post-rules）**：模型输出后执行，例如脱敏、强制补充免责声明。
- **策略优先级**：规则命中时可以直接短路（不调用模型），这是成本低、确定性高的分支。

## 案例设计

1）用户输入若包含敏感关键词，直接拒绝（不调用 LLM）。  
2）否则在用户消息前附加一条「预算约束」规则（教学展示）。  
3）模型返回后做简单后置清洗（例如去掉多余空白）。

无 Key 时仍复用与第 5 小时类似的 `mock_choice` 工具链，但会先经过规则层。

## 代码讲解

文件：`src/main.py`。

- **`RuleEngine.apply_pre(user_text) -> Result`**：返回是否允许继续、以及可能被改写后的用户文本。
- **`RuleEngine.apply_post(assistant_text)`**：返回清洗后的文本。
- **`guarded_run_session(user_goal)`**：先 `apply_pre`，再进入工具循环，最后对 `assistant.content` 做 `apply_post` 演示。

## 运行方式

```bash
cd "第6小时:规则系统"
python src/main.py
```

可尝试修改 `user_goal` 中包含敏感词以观察短路行为。
