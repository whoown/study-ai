# 第 16 小时：规划执行

## 本章目标

- 理解 **Plan-and-Execute**：先把目标拆成有序子任务（Plan），再逐步执行并更新状态（Execute）。
- 学会观察 **计划表、当前步骤、任务状态机** 的打印输出，而不是只关心最终文本。
- 体会 **失败重试与跳过策略** 对健壮 Agent 的意义（本课用模拟工具结果演示）。

## 核心概念

| 概念 | 说明 |
|------|------|
| Plan | 结构化任务列表，每步含 id、标题、依赖、状态。 |
| TaskState | 运行时快照：当前步骤、已完成集合、日志、错误计数。 |
| Executor | 接收一步计划，调用「工具」并返回观测结果（本课为函数模拟）。 |

## 案例设计

目标：「为即将发布的版本准备发布说明素材」。规划器产出 4 步：

1. 收集变更摘要（模拟成功）
2. 生成对外公告草稿（模拟成功）
3. 检查敏感词（模拟失败一次再成功）
4. 汇总输出（依赖前几步产物）

运行时在终端逐步打印：

- **`[plan]`**：完整计划 JSON
- **`[execute]`**：当前执行哪一步
- **`[task_state]`**：`current_step_id` / `completed` / `artifacts` / `errors`

## 代码讲解

代码位于 `src/main.py`：

- **`build_plan`**：返回 `list[dict]`，字段含 `id`、`title`、`depends_on`，展示 **DAG 式依赖** 的最小子集。
- **`TaskState`**：`dataclass`，集中保存可被序列化/持久化的运行时字段（为第 18 小时埋伏笔）。
- **`can_run_step`**：检查依赖是否完成，体现 **拓扑顺序** 思想。
- **`fake_tool_call`**：按步骤 id 模拟外部工具延迟、失败与返回 payload。
- **`run_plan_and_execute`**：主循环，逐步推进并在每步打印 **plan 视图 + state 视图**。

请对照输出中的 `task_state` 理解：**计划是静态蓝图，state 是动态真相**。

## 运行方式

```bash
python "第16小时:规划执行/src/main.py"
python run_demo.py 16
```

无需 API Key；全部为本地模拟执行。
