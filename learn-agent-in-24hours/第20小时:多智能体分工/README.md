# 第 20 小时：多智能体分工

## 本章目标

- 在 **第 19 小时** 的基础上，把「一个调度器 + 多个技能」推进为 **多个角色（智能体）各自负责一段推理或产出**。
- 理解 **分工**：每个角色有清晰输入/输出边界，通过共享状态（字典）传递中间结果。
- 读懂 `src/main.py` 里三个函数式智能体：`planner_agent`、`researcher_agent`、`writer_agent`。

## 核心概念

- **角色 = 智能体雏形**：本章用普通 Python 函数表示不同角色；若配置了模型，可由模型生成该角色的输出，否则使用确定性模板，保证可运行。
- **共享状态**：`AgentState`（`TypedDict`）在角色之间传递，避免隐式全局变量，便于以后接入 LangGraph。
- **与第 19 小时的关系**：第 19 小时强调「能力模块化」；本章强调「**谁**来调用哪段能力」——先把职责切开，下一小时再谈「如何编排执行顺序」。

## 案例设计

任务：给定一个用户主题，依次由 **规划员** 列提纲、**调研员** 填充要点、**写作者** 生成短文。  
全流程可在无 API Key 下完成；有 Key 时每个角色可调用一次轻量补全，以体会「多角色 + 多轮模型」的感觉。

## 代码讲解

入口文件：`src/main.py`。

- **`AgentState`**：工作流共享状态，字段包括 `topic`、`outline`、`notes`、`article`。
- **`planner_agent`**：根据 `topic` 写入 `outline`（字符串）。
- **`researcher_agent`**：读取 `outline`，写入 `notes`。
- **`writer_agent`**：综合 `topic`、`outline`、`notes`，写入 `article`。
- **`run_pipeline`**：按固定顺序调用三个智能体并打印每一步产物，体现 **线性分工**。
- **`maybe_llm_role`**：内部辅助函数，在可用模型时为某一角色生成文本，否则返回 `None` 以触发模板降级。

## 运行方式

```bash
cd "第20小时:多智能体分工"
python src/main.py
```

可选：在 `learn-agent-in-24hours/.env` 中配置 `OPENAI_API_KEY` 等，观察模型参与多角色时的输出变化。未配置时终端会提示并继续使用模板输出。
