# 第 4 小时：函数调用

## 本章定位

用结构化协议替代文本解析来描述工具调用。上一章通过正则表达式从模型的自由文本中提取 Action，这种方式脆弱且容易出错。本章引入 OpenAI 的 Function Calling 机制，让模型直接输出结构化的 JSON 来指定要调用的工具和参数，从根本上解决了"解析不可靠"的问题。这是 Agent 从实验原型走向工程化的关键一步。

## 核心概念

### Function Calling 协议是什么

Function Calling 是 OpenAI 在 2023 年引入的一项 API 能力，随后被各大模型厂商广泛采纳。它的核心思想是：**不再让模型用自然语言描述要调用什么工具，而是让模型直接输出结构化的 JSON 数据来指定工具名和参数**。

在 Function Calling 出现之前，Agent 调用工具的标准流程是：通过 Prompt 告诉模型可用的工具列表和输出格式要求 → 模型生成包含工具调用信息的文本 → 开发者用正则或字符串解析提取工具名和参数。这个流程的薄弱环节在于"文本解析"——模型的输出是非结构化的自然语言，任何微小的格式偏差都可能导致解析失败。

Function Calling 将工具描述从 Prompt 文本中抽离出来，放到 API 请求的专用字段（`tools` 参数）中。模型经过专门的微调，能够理解这些工具描述，并在需要时生成符合 JSON Schema 的调用请求。API 返回的不再是一段需要解析的文本，而是一个结构化的 `tool_calls` 对象，直接包含了工具名和参数字典。

这种协议级别的改变带来了三个核心好处：**可靠性**（JSON 格式的解析远比自由文本可靠）、**类型安全**（参数类型在 Schema 中明确定义）、**生态统一**（所有遵循协议的工具和模型可以互操作）。目前 Function Calling 已经成为 Agent 开发的事实标准。

### JSON Schema 描述工具的方式

JSON Schema 是 Function Calling 协议中用来描述工具"长什么样"的语言。每个工具通过一个 JSON Schema 对象来声明自己的名称、功能描述、参数列表、参数类型和必填项。模型根据这些 Schema 来理解每个工具的用途和调用方式。

一个工具的 Schema 通常包含以下关键部分：`name`（工具名，通常是函数名）、`description`（功能描述，这是模型理解工具用途的主要依据）、`parameters`（参数 Schema，描述每个参数的名称、类型、含义和约束）。其中 `description` 字段的质量对模型的工具选择准确性有决定性影响——它本质上是给模型看的"工具说明书"。

Schema 的设计有几个最佳实践。首先，`description` 应当清楚说明工具"做什么"而不是"怎么做"，让模型能在语义层面理解工具的适用场景。其次，参数的 `description` 应当说明参数的业务含义，而不只是技术类型信息。比如 `"查询的商品名称"` 比 `"字符串类型的参数"` 对模型更有帮助。最后，合理使用 `enum` 约束参数值范围，可以大幅减少模型传入无效参数的概率。

JSON Schema 还支持 `required` 字段来标记必填参数和可选参数。这个区分很重要：必填参数意味着模型必须提供值，可选参数意味着模型可以省略（使用默认值）。设计时应当尽量减少必填参数的数量，让工具的调用门槛更低。

### 结构化调用 vs 文本解析的优劣

这两种方式在 Agent 开发中各有适用场景，理解它们的优劣势有助于做出正确的技术选择。

文本解析（即上一章的方式）的优势在于**灵活性和模型无关性**。任何能生成文本的模型都可以使用这种方式，不需要模型支持特定的 API 协议。同时，文本格式可以完全自定义，不受协议约束。它的劣势同样明显：解析逻辑脆弱、容错成本高、维护困难。当工具数量增多或参数结构变复杂时，文本解析的工程成本会急剧上升。

结构化调用（Function Calling）的优势是**可靠性和标准化**。模型经过专门训练来生成符合格式的输出，解析只需要反序列化 JSON，几乎不存在格式问题。标准化的协议也意味着工具定义可以在不同项目间复用。它的劣势在于**依赖模型支持**——并非所有模型都支持 Function Calling，且不同厂商的协议实现可能存在细微差异。

从趋势来看，Function Calling 正在成为主流标准。但在某些场景下（如使用不支持 Function Calling 的开源模型，或需要极度灵活的自定义调用格式），文本解析仍然是必要的技能。真正成熟的 Agent 框架通常会同时支持两种方式，根据底层模型的能力自动选择。

### 参数校验（Validation）的重要性

即使使用了 Function Calling，模型生成的参数也不一定是合法的。模型可能传入类型错误的值（比如字符串代替数字）、缺失必填参数、或者传入超出范围的值。参数校验是工具执行前的最后一道防线，它确保只有合法的输入才会被传给工具函数。

参数校验在 Agent 系统中的重要性常常被低估。在传统 Web 开发中，输入校验主要是为了防止恶意用户攻击；而在 Agent 系统中，校验的主要对象是 LLM——一个"善意但不完美"的调用者。LLM 不会故意传入恶意数据，但它可能因为"理解偏差"或"幻觉"生成不合法的参数。

校验策略通常分为几个层次。第一层是**类型校验**：检查参数是否符合 Schema 中定义的类型（string、number、boolean 等）。第二层是**约束校验**：检查值是否在允许范围内（如 enum 约束、最大/最小值、字符串长度限制）。第三层是**业务校验**：检查参数在业务逻辑上是否合理（如商品数量不能为负数、日期不能在未来等），这一层通常在工具函数内部实现。

校验失败后的处理策略同样重要。最简单的方式是将校验错误作为 Observation 返回给模型："参数 X 应该是数字，但你传入了字符串"。模型通常能根据错误信息修正参数并重试。这种"校验—反馈—重试"的循环是 Agent 系统自我纠错能力的重要体现。更激进的策略包括自动类型转换（把字符串 "42" 转为数字 42）和参数默认值填充，但这些策略需要谨慎使用，以免掩盖真正的逻辑错误。

## 新手最容易卡住的点

- **Schema 的 description 写得太敷衍**：模型选择工具主要依赖 description，如果描述不清楚，模型会选错工具或传错参数。花时间写好描述的投入产出比非常高。
- **混淆 `tools` 和 `functions` 参数**：OpenAI API 经历了从 `functions` 到 `tools` 的演进。当前版本应使用 `tools` 参数。如果参考旧教程用了 `functions`，可能会遇到兼容性问题。
- **忽略模型返回的 `tool_calls` 结构**：Function Calling 的返回不在 `content` 字段中，而在 `tool_calls` 字段中。如果你习惯了从 `content` 中提取信息，需要调整解析逻辑。
- **缺少参数校验就直接执行工具**：模型生成的参数可能不合法，不做校验直接执行可能导致工具函数抛出异常或产生错误结果。

## 建议动手实验

- 修改一个工具的 `description`，使其描述模糊甚至误导性的，观察模型是否还能正确选择和调用该工具。
- 在 Schema 中添加一个 `enum` 类型的参数约束，测试模型是否遵守限制。
- 在 `run_tool` 中添加参数校验逻辑，当模型传入非法参数时返回错误信息，观察模型能否自动修正。
- 对比开启和关闭 mock 分支时的行为差异，理解 `mock_choice` 在开发调试中的作用。
